## Codebase Patterns
- HTTP route handlers are defined as standalone functions in `platform/mcp-server/src/http-routes.ts` and wired into the router via `createHandleFetch`
- Each route handler follows the pattern: check auth first, then process the request
- Route tests use `createTestHandlers()` to get handlers + state, then construct `Request` objects and call `handlers.fetch(req, mockServer)`
- Plugin tools are annotated via `state.registry.toolLookup`, browser tools via `state.cachedBrowserTools`, platform tools via `PLATFORM_TOOL_NAMES`
- `getAllToolsList(state)` from `mcp-setup.ts` returns the combined list of plugin + browser + platform tools
- `buildRegistry([plugins], [failures])` creates a plugin registry for testing
- `fetchJson<T>(handlers, url, headers?)` is a test utility for typed JSON responses
- CLI commands follow the pattern: `registerXxxCommand(program: Command)` function that creates subcommands
- CLI commands use `readAuthSecret()`, `resolvePort(options)`, `DEFAULT_HOST`, `isConnectionRefused()` for server communication
- The `fetchTools()` helper in `tool.ts` is reusable for future `tool schema` and `tool call` subcommands
- MCP gateway at `/mcp/gateway` uses separate `gatewayTransports`/`gatewaySessionServers` from full `/mcp` endpoint
- `RouteDeps` interface defines all dependencies injected into route handlers; update both `RouteDeps`, `HotState`, and `createTestHandlers` when adding new endpoint state
- `annotateTools(state)` in `mcp-gateway.ts` is a reusable function for annotating tools with their plugin names
- Plugin tools with default 'off' permission get a `[Disabled]` prefix in their description from `getAllToolsList`
- Browser tool dispatch tests need real `CachedBrowserTool` objects with Zod `input` schemas and mock `handler` functions
- `handleBrowserToolCall` validates args through the tool's Zod schema before calling the handler
- Dev proxy (`dev-proxy.ts`) must preserve the original URL path in `ProxySession.path` for all forwarding — hardcoding `/mcp` breaks gateway sessions
- E2E gateway tests need a standalone MCP client targeting `/mcp/gateway` since `createMcpClient` from fixtures.ts hardcodes `/mcp`
