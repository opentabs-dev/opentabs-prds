## Codebase Patterns
- `setupToolTest(mcpServer, testServer, extensionContext, mcpClient)` in e2e/helpers.ts handles: waitForExtensionConnected, waitForLog('tab.syncAll received'), open test tab, waitForToolResult until ready. Returns the page handle.
- After hot reload, the correct wait sequence is: waitForLog('Hot reload complete') → waitForExtensionConnected → waitForLog('tab.syncAll received') → waitForToolResult(mcpClient, 'e2e-test_echo', ..., {isError: false}).
- A second MCP client is created with `createMcpClient(mcpServer.port, mcpServer.secret)` then `await client2.initialize()`. After hot reload, `callTool` auto-reinitializes the session internally.
- Tests using fixture-based setup that exercise the extension WebSocket relay need `test.describe.serial` to avoid browser context conflicts.
- `waitForToolResult` must be imported from `./helpers.js` separately — it is not currently imported in dev-proxy.e2e.ts by default.
- For rapid successive hot reloads, ECONNREFUSED is an expected log message (proxy forwarding to already-killed worker) — do not assert its absence in those tests. Only check for 'deadlock', 'state corruption', and 'uncaughtException'.
- When firing multiple rapid triggerHotReload() calls, clear `mcpServer.logs` first; then only ONE 'Hot reload complete' log will appear (from the final worker that wins the race).
- Side panel hot reload tests (dev proxy restart) do NOT need `waitForLog(server, 'Config watcher: Watching')` — that's only needed when testing the config watcher reload path. For dev proxy hot reload, use: waitForLog('Hot reload complete') → waitForExtensionConnected → then side panel toBeVisible with 30s timeout.
- Side panel tests that use manual server setup (not fixture-based) should use `test.describe.serial` to avoid parallel context conflicts with other side panel tests in the same file.
