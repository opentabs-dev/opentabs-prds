## Codebase Patterns
- Config schema types (ConfigSettingDefinition, ConfigSchema, ConfigSettingType) are defined in platform/shared/src/manifest.ts and re-exported from platform/shared/src/index.ts
- parsePluginPackageJson validates configSchema with full field-level validation before urlPatterns validation, enabling the relaxation logic
- ConfigStatePlugin in index.ts uses `import('./manifest.js').ConfigSchema` for the configSchema field type to avoid circular re-exports
- The VALID_SETTING_TYPES set in manifest.ts is the source of truth for allowed setting types: url, string, number, boolean, select
- PluginManifestOutput in build.ts is the shape of dist/tools.json — add new top-level fields here alongside sdkVersion, icons, and tools
- validatePlugin and generateManifest in build.ts are exported via a bottom-of-file `export {}` block and tested in build.test.ts
- OpentabsConfig in config.ts has settings, localPlugins, and permissions — all three must be preserved in savePluginPermissions
- configSchema flows: package.json → shared parsing → plugin-tools build (dist/tools.json) → loader.ts → RegisteredPlugin → discovery settings resolution
- Settings resolution (settings-resolver.ts) runs between Phase 4 (merge) and Phase 5 (buildRegistry) in discovery.ts
- url-type settings derive Chrome match patterns via `*://hostname/*` and populate homepage if static homepage is undefined
- Settings are injected into tab MAIN world before the adapter IIFE via injectPluginConfig in iife-injection.ts, setting globalThis.__openTabs.pluginConfig
- resolvedSettings flows: server buildConfigStatePayload/resolvePluginSettings → sync.full/plugin.update → extension validatePluginPayload → PluginMeta storage → injectAdapterFile → MAIN world
- PluginState in bridge.ts inherits configSchema and resolvedSettings from ConfigStatePlugin via Omit<ConfigStatePlugin, 'tools'> — no explicit type changes needed
- PluginMenu uses an onConfigOpen callback prop from PluginCard (which owns the ConfigDialog state) to trigger the settings dialog
- needsSetup helper is exported from ConfigDialog.tsx and reused in PluginCard for badge visibility and content area switching
- ConfigDialog uses uncontrolled form pattern (defaultValue + formRef) for commit-on-save behavior — avoids fighting user input with React re-renders
- Radix Select requires a name prop for form access (unlike Input which uses native form.elements.namedItem)
- Switch (boolean) fields use data-key attribute and data-state="checked" for form reading since Switch is not a native form element
- Side panel → background message relay handlers live in background-message-handlers.ts (not background.ts) and use sendServerRequest for JSON-RPC relay
- New bg:* message types need: interface in extension-messages.ts, handler in background-message-handlers.ts, entry in backgroundHandlers map, entry in EXTENSION_ONLY_TYPES, and export
- pluginNameFromPackage derives the short plugin name used as the settings key — it strips PLUGIN_PREFIX and @opentabs-dev scope
- CLI plugin discovery (findPluginDir) checks local plugins first, then global npm via resolvePluginPackageCandidates candidates
- Settings in config.json use the short plugin name as key (e.g., settings.slack not settings.@opentabs-dev/opentabs-plugin-slack)
