## Codebase Patterns
- `ws` package is a root workspace dependency — importable in e2e/test-server.ts without adding to e2e package.json
- ESLint import-x/order groups: `['index', 'sibling', 'parent', 'internal', 'external', 'builtin', 'object', 'type']` — external (ws) before builtin (node:http), all `import type` last
- For ws RawData: use `Buffer.isBuffer(data) ? data.toString('utf-8') : Array.isArray(data) ? Buffer.concat(data).toString('utf-8') : Buffer.from(data).toString('utf-8')` to avoid @typescript-eslint/no-base-to-string
- WebSocket server created with `new WebSocketServer({ server, path: '/ws' })` to share the same HTTP port
- WebSocket HTML page uses `'ws://' + location.host + '/ws'` for the connection URL
- Adding a new browser tool requires updating 8+ files: network-capture.ts (CDP events + public API), network-commands.ts (handler), browser-commands/index.ts (export), message-router.ts (import + dispatch), known-methods.ts (DISPATCH_METHODS), browser-tool-names.ts (server), browser-tools/index.ts (barrel), e2e/helpers.ts (BROWSER_TOOL_NAMES), and message-router.test.ts + known-methods.test.ts (mocks)
- After modifying extension source, `npm run build:force` is needed (not just `npm run build`) because tsc incremental compilation may not rebuild esbuild bundles with new code
- Extension tools dispatch via `browser.<methodName>` JSON-RPC methods; sync handlers use wrapSync, async use wrapAsync in message-router.ts
- CDP WebSocket events: webSocketCreated (provides requestId + url mapping), webSocketFrameSent/Received (provides frame payload via response.payloadData), webSocketClosed (cleanup)
- Adding a HAR export tool follows the same 8+ file pattern as other browser tools — network-capture.ts (public API), network-commands.ts (handler), barrel, router, known-methods, tool-names, MCP definition, E2E helpers, test mocks
- HAR 1.2 conversion: map CapturedRequest fields to HAR entry structure. Use `new URL(req.url).searchParams` for query string parsing. Headers are scrubbed before export.
- The `exportHar()` function in network-capture.ts is a sync function (no CDP calls) that converts the in-memory request buffer to HAR JSON, so the handler uses wrapSync in the dispatch table
- Chrome's CDP does NOT expose `Cookie` request headers or `Set-Cookie` response headers in Network events (privacy/security restriction). Test header scrubbing with explicitly-set custom headers (X-Api-Key, X-Auth-Token, etc.) which CDP does report
- CDP `Runtime.consoleAPICalled` uses `type = 'warning'` for `console.warn` (not 'warn'). Stored level in consoleLogs is 'warning'. The MCP level filter with `level='error'` filters using exact match against stored levels
- The waitForConsoleLogs helper accepts a predicate and returns all logs (not just matching entries) — use it to wait, then find specific entries manually
