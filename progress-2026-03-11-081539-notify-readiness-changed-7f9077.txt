## Codebase Patterns
- The plugin SDK uses `globalThis.__openTabs` as a runtime bridge between the IIFE wrapper and SDK functions. The IIFE wrapper sets closures/nonces on this object, and SDK functions read them.
- The log transport pattern (`log.ts`) is the template for cross-world communication: MAIN world posts via `window.postMessage` with a nonce, ISOLATED world validates the nonce and forwards via `chrome.runtime.sendMessage`.
- SDK functions that only work in browser context must be wrapped in try/catch and return silently in Node.js environments (no `window`, no `globalThis.__openTabs`).
- The background `MessageHandler` type receives only `(message, sendResponse)` â€” `sender` from `chrome.runtime.onMessage` is not passed through. Content-script-origin messages (like `plugin:logs`, `plugin:readinessChanged`, `tool:progress`) must NOT be added to `EXTENSION_ONLY_TYPES`.
- `notifyAffectedPlugins` takes `PluginMeta[]` and handles full diff detection + notification. Use `getPluginMeta(name)` to look up metadata from a plugin name string.
- The `npm run test:e2e` script does `npm ci` in `plugins/e2e-test/` which reinstalls deps from npm. Unpublished platform changes (plugin-sdk, plugin-tools) are NOT available in the rebuilt plugin. The IIFE wrapper code comes from published plugin-tools, but the browser extension relay code uses local builds. To test features needing unpublished SDK exports, use `globalThis.__openTabs` directly or post raw `window.postMessage` with the nonce.
- `validate-icon.ts` uses `MAX_ICON_SIZE = 16 * 1024` (16KB) â€” E2E icon tests must match this threshold.
