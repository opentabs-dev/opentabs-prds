## Codebase Patterns
- url-type config field values are now `Record<string, string>` (instance name â†’ URL) in config.json, not plain strings
- Two paths set plugin settings: HTTP `POST /plugin-settings` (http-routes.ts, CLI) and WebSocket `config.setPluginSettings` (extension-handlers.ts, side panel) â€” both must validate consistently
- Plugin configSchema is accessed via `state.registry.plugins.get(pluginName)?.configSchema`
- resolvedValues type is `Record<string, unknown>` (not `Record<string, string | number | boolean>`) â€” url fields store maps, other fields store primitives
- The resolvedSettings type flows through: settings-resolver.ts â†’ ConfigStatePlugin (shared) â†’ PluginMeta (extension-messages.ts) â†’ ValidatedPluginPayload + ServerOnlyPluginFields (message-router.ts) â†’ ConfigDialog props â†’ injectPluginConfig. Changing the type requires updating all these.
- parseResolvedSettings (message-router.ts) validates and passes through both primitives and plain objects (url field maps)
- ResolvedSettings.instanceMap maps instance name â†’ Chrome match pattern, built from all url-type config fields
- Chrome match patterns from deriveMatchPattern are always `*://hostname/*` â€” hostname extraction via regex `/^\*:\/\/([^/]+)\/\*$/` is reliable
- getMergedTabMapping merges tab mappings from all extension connections â€” use for cross-connection tab lookups
- Platform-injected params (tabId, instance) are extracted via destructuring before Ajv validation and stripped from plugin args
- Per-tab config resolution: `resolvePerTabSettings` in iife-injection.ts matches tab hostname against instanceMap patterns and replaces url field maps (Record<string, string>) with the single URL string for that tab's instance
- instanceMap propagation in extension: 4 code paths must pass it: sync.full, plugin.update, tab.onUpdated (injectPluginsIntoTab), and extension startup (reinjectStoredPlugins)
- `queryMatchingTabs` returns `{tabId, url}` pairs; `queryMatchingTabIds` is a thin wrapper returning just IDs
- For multi-instance E2E tests, use localhost vs 127.0.0.1 as distinct instances since `deriveMatchPattern` strips ports (both `http://localhost:3000` and `http://localhost:4000` derive `*://localhost/*`)
