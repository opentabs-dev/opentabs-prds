## Codebase Patterns
- `reload.ts` has two reload paths: `performReload` (full hot reload, also used at startup) and `performConfigReload` (config-only reload, used by POST /reload, file watcher, plugin install/uninstall)
- `reloadCore` is the shared fast state-swap path — update checks and extension notifications belong in the callers, not in reloadCore
- `buildConfigStatePayload` and `sendToExtension` are in `extension-handlers.ts`; `isExtensionConnected` is in `state.ts`; `checkForUpdates` is in `version-check.ts`
- Imports in reload.ts must be alphabetically sorted (biome enforces this)
- The `plugins.changed` notification format: `{ jsonrpc: '2.0', method: 'plugins.changed', params: { ...buildConfigStatePayload(state) } }`
- To mock an async function in vitest: use `vi.mock('./module.js')` at module level (hoisted), import the function, then use `vi.mocked(fn).mockResolvedValue(undefined)` in beforeEach. The mock replaces the module's export codebase-wide in that test file.
- `await undefined` is valid JS/TS — `vi.fn()` returns `undefined` by default, so existing tests using `await asyncMockedFn()` won't break even without `mockResolvedValue`
- `checkForUpdates` returns early (`if (npmPlugins.length === 0) return;`) for local-only plugin setups WITHOUT touching `state.outdatedPlugins` — PRD notes claiming it "replaces with empty array" are incorrect. Local plugins have `source: 'local'` and are filtered out, so `npmPlugins.length === 0`.
- ALL plugins (including local ones) have `npmPackageName` set to `pkg.name` from their `package.json` (see `loader.ts:432`). So `pruneStaleState` will NOT prune injected outdated data for local plugins, since `npmPkgNames` includes the local plugin's package name.
- For E2E tests, POST /reload preserves injected outdated data for local plugins (dot stays visible), because: pruneStaleState keeps it, sendSyncFull includes it, checkForUpdates returns early (no change), plugins.changed is sent (but same data). The dot stays visible.
