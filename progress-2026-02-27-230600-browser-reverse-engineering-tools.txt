## Codebase Patterns
- `network-capture.ts` manages per-tab capture state via a `Map<number, CaptureState>`. All state is cleared by `captures.delete(tabId)` in both `stopCapture` and cleanup listeners.
- The Chrome extension uses `chrome.debugger.onEvent.addListener` (registered once at module load) for all CDP event handling — add new event handlers in the existing if/else chain.
- The `import-x/exports-last` ESLint rule requires `export interface` / `export type` to be moved to standalone `export type { ... }` statements at the bottom of the file, not declared inline with `export interface`.
- Exports in the browser-extension follow the pattern: inline `export const`/`export function` for values, bare `export { name }` and `export type { TypeName }` for re-exports at the bottom.
- CDP WebSocket frame events: `Network.webSocketFrameSent` and `Network.webSocketFrameReceived` both carry `{ requestId, response: { opcode, mask, payloadData } }`. The `requestId` maps to the WS URL from `Network.webSocketCreated`.
- Adding a new browser method requires updates in 5 places: (1) browser-commands handler, (2) browser-commands/index.ts export, (3) message-router.ts dispatch table, (4) known-methods.ts DISPATCH_METHODS, (5) tests in both message-router.test.ts and known-methods.test.ts.
- Import ordering in browser-tools/index.ts is alphabetical by filename — `get-websocket-frames.js` goes after `get-tab-info.js` (w > t).
- Adding a new MCP browser tool requires updates in 5 places: (1) new tool file in browser-tools/, (2) browser-tools/index.ts (import + array), (3) browser-tool-names.ts, (4) permissions.ts TOOL_TIERS, (5) extension handler.
- Server-side-only browser tools (e.g., `browser_export_har`) don't need a new extension dispatch method — they reuse existing dispatch methods and do conversion server-side. Only 4 places need updating: (1) new tool file, (2) index.ts barrel, (3) browser-tool-names.ts, (4) permissions.ts.
- When adding new browser tools, the E2E helpers file `e2e/helpers.ts` has a separate `BROWSER_TOOL_NAMES` array that must also be updated — it's separate from `browser-tool-names.ts` and the E2E tests fail if it's out of sync.
- `detect-apis.ts` accepts optional `WsFrame[]` as second argument to `detectApis()` for enriching WebSocket endpoint detections with frame samples.
