## Codebase Patterns
- `platform/shared/src/generated/browser-tools-catalog.ts` is auto-generated by `scripts/generate-browser-tools-catalog.ts` during every `npm run build`. Do NOT manually edit the generated file without also updating the generator script — the build will overwrite your changes.
- The generator script generates both the `BrowserToolMeta` interface template AND the catalog entries. Changes to the interface must be made in the `lines` array in the generator script (around line 58-62).
- Optional fields in type interfaces use the pattern `...(x ? { field: x } : {})` for conditional inclusion in object literals throughout the codebase.
- `CachedBrowserTool` in `platform/mcp-server/src/state.ts` mirrors `BrowserToolDefinition` but without the handler; it's the pre-computed version stored in server state.
- `ConfigStateBrowserTool` in `platform/shared/src/index.ts` is the wire type sent to the browser extension side panel via `config.getState` WebSocket response.
- The barrel validation in `platform/mcp-server/src/browser-tools/index.ts` runs at module import time. When adding a new validated field, the catalog must already contain the field before the barrel can be imported. The generator imports the barrel, creating a chicken-and-egg problem. Workaround: temporarily disable the new validation in the compiled JS (`dist/`), run the generator, then rebuild from source.
- Threading a new optional field through the full stack requires changes in 5 locations: `rebuildCachedBrowserTools` (mcp-setup.ts), `buildConfigStatePayload` (extension-handlers.ts), generator script type cast + extraction + emission, barrel validation, and the generated catalog file.
