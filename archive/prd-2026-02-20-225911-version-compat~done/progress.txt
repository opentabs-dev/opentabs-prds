## Codebase Patterns
- The `PluginManifestOutput` interface in `platform/plugin-tools/src/commands/build.ts` defines the shape of `dist/tools.json` — add new fields there and in the shared `PluginManifest` type in `platform/shared/src/index.ts`
- Plugin dist files (`plugins/*/dist/`) are gitignored and not part of the monorepo build — they must be updated separately
- The build-e2e tests in `platform/plugin-tools/src/commands/build-e2e.test.ts` run the local compiled build command (not the published npm version), so they exercise unreleased changes
- The `resolveSdkVersion()` function reads from `node_modules/@opentabs-dev/plugin-sdk/package.json` relative to the plugin project directory
- `generateManifest()` takes an `sdkVersion` parameter and places it as the first field in tools.json
- The server's SDK version is resolved from `platform/plugin-sdk/package.json` (relative path from mcp-server), following the same `import.meta.url` + dirname pattern used by `version.ts`
- `LoadedPlugin` and `RegisteredPlugin` both carry `sdkVersion?: string` — flows through from loader → registry unchanged
- `checkSdkCompatibility()` is a pure function (no side effects) — easy to unit test with arbitrary version pairs
- For 0.x versions (e.g., 0.0.16), major=0 and minor=0 — the patch version is NOT compared, so all 0.0.x plugins are mutually compatible
- The `/health` endpoint returns `sdkVersion` at top level (server's version) and per plugin in `pluginDetails[]` (null if missing)
- `config.getState` includes `sdkVersion` per plugin via `ConfigStatePlugin` type (optional field, spread conditionally)
- Side panel `PluginCard` uses `Tooltip.Provider`/`Tooltip` from the retro Tooltip component — must be self-contained (Provider inside the trigger area)
- E2E types in `e2e/fixtures.ts` (HealthResponse, PluginDetail) must be kept in sync with actual `/health` endpoint response structure
- Tests needing custom plugin configs use `copyE2eTestPlugin()` + standalone `startMcpServer()` (not fixtures) for isolation
