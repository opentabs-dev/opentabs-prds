## US-006: Offscreen document listens for port changes and reconnects ✅

Commit: e2009d7

Changes:
- platform/browser-extension/src/extension-messages.ts: Added `PortChangedMessage` interface
  (`{ type: 'port-changed', port: number }`) and included it in the `InternalMessage` union.
- platform/browser-extension/src/background.ts: Added `port-changed` case to the message
  listener that relays the message to the offscreen document via `chrome.runtime.sendMessage`.
- platform/browser-extension/src/offscreen/index.ts: Added `port-changed` case to the message
  listener. Constructs `ws://localhost:{port}/ws` URL, compares to current `mcpServerUrl`, and
  if different: updates the URL, resets backoff, closes existing WebSocket (or cancels pending
  reconnect), and initiates immediate reconnect.

All checks pass: build, type-check, lint, knip, test (1789), test:e2e (340).

**Learnings for future iterations:**
- The offscreen document already had two mechanisms for URL changes: `ws:setUrl` (explicit
  message with full URL) and the storage change listener in background.ts that converts
  `serverPort` changes to `ws:setUrl`. The `port-changed` message provides a third, more
  direct path from the side panel that doesn't depend on storage change listeners firing.
- The `port-changed` handler in offscreen follows the same close-and-reconnect pattern as
  `ws:setUrl` but is simpler since it only needs to construct the URL from a port number
  rather than validate an arbitrary URL.
- The background script acts as a relay for `port-changed` because `chrome.runtime.sendMessage`
  in the side panel context can reach both the background and offscreen listeners. The background
  forwards it to ensure the offscreen document receives it even if the side panel's direct
  message doesn't reach the offscreen context.

## US-008: Side panel: distinct error states for connection refused vs auth failure ✅

Commit: 62524cb

Changes:
- platform/browser-extension/src/extension-messages.ts: Added `DisconnectReason` type
  (`'connection_refused' | 'auth_failed'`). Extended `WsStateMessage` with optional
  `disconnectReason` field. Extended `SpConnectionStateMessage.data` with optional
  `disconnectReason`.
- platform/browser-extension/src/offscreen/index.ts: Added `lastDisconnectReason` tracking.
  `refreshWsUrl` now returns `DisconnectReason | undefined` — returns `'auth_failed'` when
  /ws-info returns 401 after retry, `'connection_refused'` on network error. `connect()`
  propagates the reason via `ws:state` messages. `onclose` handler detects WS close code 4001
  as auth failure. `ws:getState` response includes `disconnectReason`.
- platform/browser-extension/src/background.ts: Added `lastDisconnectReason` tracking.
  `ws:state` handler stores and forwards the reason to the side panel via `sp:connectionState`.
  `bg:getConnectionState` handler returns `disconnectReason` alongside `connected`.
- platform/browser-extension/src/side-panel/bridge.ts: `getConnectionState` now returns
  `ConnectionStateResult` (`{ connected, disconnectReason? }`) instead of `boolean`.
- platform/browser-extension/src/side-panel/App.tsx: Added `disconnectReason` state. Reads it
  from initial `getConnectionState()` call and from `sp:connectionState` messages. Passes it
  to `DisconnectedState` via `reason` prop.
- platform/browser-extension/src/side-panel/components/EmptyStates.tsx: Split `DisconnectedState`
  into two sub-components: `ConnectionRefusedState` (shows "Cannot Reach MCP Server" with
  `opentabs start --port <N>` where N is from chrome.storage.local) and `AuthFailedState`
  (shows "Authentication Failed" with instructions to reload from chrome://extensions/).
- platform/browser-extension/src/side-panel/bridge.test.ts: Updated `getConnectionState` tests
  to expect `ConnectionStateResult` objects instead of booleans.

All checks pass: build, type-check, lint, knip, test (1789), test:e2e (339 + 1 flaky).

**Learnings for future iterations:**
- The disconnect reason flows through three layers: offscreen → background → side panel.
  Each layer has its own message type (`ws:state`, `sp:connectionState`) that carries the
  optional `disconnectReason` field.
- The MCP server returns HTTP 401 for both `/ws-info` and `/ws` upgrade when the secret
  doesn't match. The primary auth failure detection happens in `refreshWsUrl()` (HTTP fetch)
  before attempting the WebSocket connection. Browsers don't expose HTTP status codes from
  failed WebSocket upgrades — the close code is always 1006 (abnormal closure).
- The `bg:getConnectionState` handler is the initial state source when the side panel opens.
  It must also return `disconnectReason` so the side panel shows the correct error state
  immediately on load (not just after receiving a `sp:connectionState` update).
- The `ConnectionRefusedState` component reads the port from `chrome.storage.local` to show
  the correct `opentabs start --port <N>` command. When port is the default (9515), it omits
  the `--port` flag for cleaner output.
