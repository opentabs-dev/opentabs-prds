## Codebase Patterns
- The MCP server logger is imported as `import { log } from './logger.js'` in platform/mcp-server/src/ files
- `log.debug()` is the correct level for per-directory/per-file failures in plugin discovery (noisy in environments with many global modules)
- `log.warn()` is appropriate for summary-level signals that users should act on (e.g., both npm and bun resolution failing)
- The `plugins/e2e-test/dist/` directory is gitignored and must be built before running `bun run test` — run `cd plugins/e2e-test && bun install --frozen-lockfile && bun run build` if the build-e2e.test.ts tests fail with ENOENT on dist/index.js
- Error message pattern in catch blocks: `e instanceof Error ? e.message : String(e)` — use this consistently for error stringification
- In build.ts, `validatePlugin` handles hard errors (empty names, bad formats), while `warnResourcesAndPrompts` handles soft warnings (wrong field types for optional fields) — don't duplicate checks across both
