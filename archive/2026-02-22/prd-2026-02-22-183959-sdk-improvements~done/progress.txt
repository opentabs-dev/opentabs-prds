## Codebase Patterns
- SDK fetch utilities in `platform/plugin-sdk/src/fetch.ts` follow a consistent pattern: interface with overloaded call signatures (schema vs no-schema), then an implementation function cast to the interface type. All use `fetchJSONImpl` for JSON parsing + optional Zod validation.
- Extra headers from `init?.headers` are extracted via `Object.fromEntries(new Headers(init.headers).entries())` so that the explicit Content-Type header takes precedence.
- The e2e-test plugin at `plugins/e2e-test/` must be built (`bun install && bun run build`) before running build-e2e tests — it's a standalone project not covered by root `bun install`.
- Discovery intentionally skips nonexistent local plugin paths (stale config entries) — they don't appear in `failedPlugins`. Only paths that exist but fail to load appear there.
- TypeScript method syntax in interfaces (e.g., `render(args: X): R`) is bivariant under `strictFunctionTypes` — typed prompt render functions are directly assignable to `PromptDefinition` without casts.
