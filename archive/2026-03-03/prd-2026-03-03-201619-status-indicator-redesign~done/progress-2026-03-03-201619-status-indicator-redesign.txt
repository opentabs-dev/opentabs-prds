## Codebase Patterns
- PluginIcon uses `useSyncExternalStore` with a MutationObserver to subscribe to dark mode class changes on `<html>` — this pattern avoids stale dark mode state in the side panel.
- CSS animations in `styles.css` use CSS custom property references (`var(--primary)`, `var(--border)`) so they automatically adapt to light/dark mode.
- The side panel components use `cn()` (a `clsx`+`twMerge` wrapper) for conditional class composition.
- Tailwind opacity modifiers like `border-border/30` work via the `@theme` bridge in `styles.css` — no custom CSS needed for opacity variants.
- E2E CSS selectors for Tailwind opacity modifiers: `.border-border\/30` (CSS) → `'.border-border\\/30'` (JS string) → `page.locator('.border-border\\/30')` (Playwright). The `/` must be escaped.
- CSS class `.border-border` does NOT match elements with class `border-border/30` — they are distinct class names. Use this to differentiate ready (`border-border`) from not-ready (`border-border/30`) states in E2E tests.
- BrowserToolsCard always uses `tabState="ready"` for its PluginIcon → its icon always has class `border-border`. When scoping E2E selectors, always use the specific plugin card button as the root to avoid false matches from the Browser card.
- Border animations (`animate-activity-border-flash/fade-out`) are applied as class toggles on always-present container elements. MutationObserver should use `attributes: true, attributeFilter: ['class']` only — NOT `childList` — since no nodes are added/removed.
