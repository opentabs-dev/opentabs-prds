## Codebase Patterns
- PluginIcon uses `getBorderClasses()` to compute border classes based on tabState, active, and fadingOut state
- The fade-out animation logic (useRef + useEffect for active→inactive transition) lives directly in the PluginIcon component
- CSS border animations (`animate-activity-border-flash`, `animate-activity-border-fade-out`) are defined in styles.css (the old dot-based `activity-flash`/`activity-fade-out` animations were removed in US-004)
- Not-ready plugins use `border-border/30` (Tailwind opacity modifier) for faded ghost borders
- The `cn()` utility from `../lib/cn.js` is used for conditional class merging
- E2E tests detect plugin readiness via `[class*="border-border/30"]` (visible = not-ready, hidden = ready); the Tailwind opacity modifier `/30` produces a CSS class with literal `/30` in the name
- E2E tests detect activity animations via `.animate-activity-border-flash` (pulsing border) instead of old `.animate-activity-flash` (opacity dot)
- BrowserToolsCard always renders with `tabState="ready"`, so `.border-border` locators will find it even when no plugin tab is open
