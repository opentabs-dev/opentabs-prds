## Codebase Patterns
- File watcher is in platform/mcp-server/src/file-watcher.ts — uses fs.watch() with 200ms debounce and mtime polling fallback
- MTIME_POLL_INTERVAL_MS (default 30s) is the fallback polling interval; configurable via OPENTABS_MTIME_POLL_MS env var
- E2E test MCP servers are spawned in e2e/fixtures.ts via startMcpServer() — all env vars for test isolation are set there
- Tests wait for specific log messages (e.g., "File watcher: Watching", "Config watcher: Watching") before modifying files
- On Linux, inotify max_user_instances (default 128) limits how many fs.watch() instances a single user can have
- Each Chromium process consumes several inotify instances; at 16 workers this exhausts the 128 limit
- When fs.watch() fails with EMFILE, the try/catch in watchPlugin() silently catches it — the watcher is never established
- The mtime poll fallback is the only mechanism that works when fs.watch is exhausted

## 2026-02-27 23:08 - US-001
- Diagnosed root cause: Linux inotify instance exhaustion (EMFILE) under PW_WORKERS=16
- /proc/sys/fs/inotify/max_user_instances=128 is the bottleneck
- With 16 Playwright workers, each spawning MCP server + Chromium, the combined inotify instance count exceeds 128
- fs.watch() fails silently (caught by try/catch), leaving watchers unestablished
- The mtime polling fallback (30s interval) is too slow for E2E test timeouts (10-15s)
- Config watcher failure prevents "Config watcher: Watching" log from being emitted, causing waitForLog() to timeout
- Files changed:
