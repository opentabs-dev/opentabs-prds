## Codebase Patterns
- Plugins in `plugins/` are standalone projects requiring separate `bun install` and `bun run build` before their tests work in the root monorepo's `bun run test`
- The `build-e2e.test.ts` in plugin-tools depends on `plugins/e2e-test/dist/` existing with built artifacts and `node_modules/` being installed
- Variable rename refactors are pure renames — no logic changes, only update variable names and their references within the same scope
- File renames in browser-extension use `.js` extensions in imports (ESM), so search for both `./types` and `./types.js` patterns
- After renaming files, check ESLint `import-x/order` rule — alphabetical sort changes when the filename changes (e.g., `./bg-log-state` → `./background-log-state` sorts differently)
- Browser-extension constants are split: core constants in `src/constants.ts`, side-panel UI constants in `src/side-panel/constants.ts`
- The IIFE wrapper in plugin-tools `build.ts` is a TypeScript template string bundled by `Bun.build()` — `declare global` augmentations work for typing `globalThis` access

- Files changed: `platform/plugin-tools/src/commands/build.ts`
- **Learnings for future iterations:**
  - The IIFE wrapper is a TypeScript template string written to a `.ts` file and bundled by `Bun.build()`. TypeScript types DO apply — Bun handles TypeScript in the wrapper file. So `declare global` augmentations work and are the correct approach for typing `globalThis` access.
  - `delete (obj as Record<string, unknown>).prop` is a cleaner alternative to `obj.prop = undefined as any` when clearing optional method properties on typed objects.
  - Using rest parameters (`...args`) is cleaner than `arguments` + `as any` for forwarding function arguments through a wrapper.
