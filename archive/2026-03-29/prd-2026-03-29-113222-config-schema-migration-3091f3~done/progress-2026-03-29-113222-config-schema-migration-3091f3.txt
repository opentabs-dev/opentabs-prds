## Codebase Patterns
- OpentabsConfig interface is in platform/mcp-server/src/config.ts — all construction sites must include every field
- savePluginPermissions and savePluginSettings build OpentabsConfig manually from `current` fields — any new field on OpentabsConfig must be added to both functions
- E2E fixtures (e2e/fixtures.ts) have their own OpentabsConfig interface with optional fields — it's separate from the server's type
- parseConfigRecord is the single parsing entry point for raw JSON → OpentabsConfig — all validation/defaults belong there
- Config test pattern: use mkdtempSync + OPENTABS_CONFIG_DIR env override for test isolation
- config-migrations.ts exports CURRENT_CONFIG_VERSION and migrateConfig — config.ts imports CURRENT_CONFIG_VERSION for the default config version
- Migrations operate on raw Record<string, unknown>, not parsed OpentabsConfig — old shapes may not match the current type
- The v1→v2 migration detects url-type fields heuristically: any string starting with http:// or https:// is wrapped as { default: value }
- Migration mutates fields in-place on the structuredClone'd config — no need to return a new settings object
- E2E tests for config migration should use raw JSON writes (not writeTestConfig) for full control over config shape
- E2E tests can verify migration effects by reading config.json from disk after server.waitForHealth resolves
