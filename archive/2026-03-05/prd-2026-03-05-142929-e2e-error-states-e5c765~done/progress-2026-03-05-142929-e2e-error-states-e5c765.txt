## Codebase Patterns
- Failed plugin data flows: discovery errors → state.discoveryErrors → buildConfigStatePayload → sync.full → side panel failedPlugins state → FailedPluginCard
- discoveryErrors has shape `{ specifier: string; error: string }[]` — specifier is the raw path from localPlugins config
- FailedPluginCard shows "Show more"/"Show less" toggle when error.length > 100
- E2E tests for side panel use manual setup (not fixtures) when custom config is needed: createTestConfigDir/writeTestConfig → startMcpServer → launchExtensionContext → setupAdapterSymlink → waitForExtensionConnected → openSidePanel
- To create a broken plugin for testing: make a directory with valid package.json (opentabs field) but missing or corrupted dist/ artifacts
- Optimistic rollback (PluginCard.tsx) cannot be triggered in E2E: server disconnect wipes all state before the rollback error can be observed. Test via component tests instead.
- Permission change flow: side panel → bridge.setToolPermission → bg:setToolPermission → sendServerRequest (WebSocket) → server config.setToolPermission → config.json write + plugins.changed notification
- `createMinimalPlugin` from fixtures creates a discoverable plugin with tools but `isReady: false` — sufficient for permission/config tests without a test server
- Disconnect/reconnect E2E pattern: save `server.port`, `server.kill()`, then `startMcpServer(configDir, true, serverPort)` to restart on the same port
- `browserToolsOpen` persists in `chrome.storage.session` across disconnect/reconnect; `searchQuery` and npm results are wiped by App.tsx disconnect handler
- Use `{ exact: true }` with `getByText()` when the text appears in both a specifier element and an error message element (error messages embed file paths)
