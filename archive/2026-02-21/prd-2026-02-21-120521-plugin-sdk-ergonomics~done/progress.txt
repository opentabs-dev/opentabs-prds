## Codebase Patterns
- Plugin SDK public API in `platform/plugin-sdk/src/index.ts` uses `import-x/exports-last` lint rule — every declaration between imports and the end of file must be exported. Non-exported interfaces break the lint rule.
- The `import-x/exports-last` rule applies to ALL SDK source files (`fetch.ts`, `storage.ts`, etc.), not just `index.ts`. Any helper function must be exported and re-exported from `index.ts`.
- For overloaded factory functions (like `definePrompt`), use an exported call signature interface + `export const` instead of `export function` with overloads, to satisfy the exports-last rule.
- Zod 4: `instanceof z.ZodOptional` and `instanceof z.ZodDefault` work for detecting optional fields. `.description` property on schemas works for extracting `.describe()` metadata.
- Plugins in `plugins/` depend on published npm versions of `@opentabs-dev/plugin-sdk`, not the source. New SDK features can only be used in plugins after publishing.
- ToolError factory method codes are uppercase (`TIMEOUT`, `NOT_FOUND`, `AUTH_ERROR`, `RATE_LIMITED`, `VALIDATION_ERROR`, `INTERNAL_ERROR`). Tests must use these exact codes, not ad-hoc lowercase ones.
- `ToolDefinition` (SDK) allows optional fields for authoring convenience; `ManifestTool` (shared) keeps them required since `opentabs-plugin build` always resolves defaults before writing `tools.json`. The loader validates `ManifestTool` fields at load time.
- Use `string.charAt(0)` instead of `string[0]` in strict TypeScript (noUncheckedIndexedAccess). Cast `ToolDefinition` to `Record<string, unknown>` via `as unknown as Record<string, unknown>`.
- The `ToolHandlerContext` with `reportProgress` is created in `platform/browser-extension/src/tool-dispatch.ts` (in `executeToolOnTab`), not in the adapter IIFE wrapper (`build.ts`). Changes to the progress context API must be made there.
- `@typescript-eslint/no-unnecessary-type-parameters` (from `strictTypeChecked`) flags return-only generics like `<T>(path: string): T | undefined`. Workaround: use a branded intersection `<T>(path: string & { __brand?: T }): T | undefined` so T appears in both parameter and return. Combine with a non-generic overload via a call signature type for backward compat.
- bun-types `expect()` overloads resolve `unknown | undefined` differently from `unknown` — a return type of `T | undefined` where `T = unknown` breaks `.toBe()` assertions that worked with plain `unknown`. Fix: use overloaded call signature types with non-generic first, generic second.
- When an overloaded function needs to be called internally (e.g., `postJSON` → `fetchJSON`), extract the core logic into a non-overloaded `*Impl` function. Overload resolution fails when an argument is `T | undefined` and neither overload signature accepts `undefined`. Both the public overloaded const and internal callers use the impl directly.
- Generic interfaces with type parameters in covariant-only positions (readonly/optional fields, return types) can be used directly in arrays (`ResourceDefinition[]`) without issues. The overloaded factory pattern is only needed when the generic appears in contravariant positions (method parameters), as with `PromptDefinition`'s `render(args)`.

- Files changed:
  - `platform/plugin-sdk/src/page-state.ts` — added `GetPageGlobal` overloaded call signature type with non-generic `(path: string): unknown` and generic `<T>(path: string & { __brand?: T }): T | undefined` overloads; changed `getPageGlobal` from plain arrow function to typed const using `GetPageGlobal`

- **Learnings for future iterations:**
  - `@typescript-eslint/no-unnecessary-type-parameters` (part of `strictTypeChecked`) flags generics that appear only once in a function signature — even in call signatures inside interfaces and type literals. The `__brand` intersection pattern (`string & { __brand?: T }`) makes T appear in both the parameter and return type positions, satisfying the rule without changing the runtime API.

When `getPageGlobal` returned `T | undefined` with `T = unknown`, `expect(result).toBe('hello')` would fail at the type level because bun-types inferred `Matchers<undefined>`. The fix: use call signature overloads with non-generic `(path: string): unknown` listed first, so calls without explicit type args match the non-generic overload and return plain `unknown`.
  - TypeScript resolves call signature overloads top-to-bottom. When an explicit type argument is provided (`getPageGlobal<string>(...)`), TS matches the generic overload. Without an explicit type argument, TS matches the first non-generic overload. Order matters.
