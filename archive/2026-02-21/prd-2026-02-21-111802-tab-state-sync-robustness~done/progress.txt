## Codebase Patterns
- Tab event listeners in background.ts follow a consistent pattern: onUpdated (inject + check), onRemoved (check with removed=true), onReplaced (remove old + inject new + check)
- The onUpdated status=complete path sequences injection before state check: `injectPluginsIntoTab().then(() => checkTabStateChanges())`
- checkTabStateChanges(tabId, undefined, true) is the pattern for marking a tab as removed
- checkTabStateChanges(tabId, { status: 'complete' }) triggers the "page finished loading" path which fetches the tab URL and filters plugins by pattern match
- Any code path that sends `tab.stateChanged` must update `lastKnownState` cache first (via `updateLastKnownState` or inline `lastKnownState.set`), or deduplication in `checkTabStateChanges` breaks
- E2E tests using the fixture approach (`test` from `fixtures.js`) get automatic setup/teardown of `mcpServer`, `testServer`, `extensionContext`, `mcpClient` — prefer this over manual setup
- `setupToolTest` from `helpers.ts` is the standard preamble for fixture-based tests that need a ready plugin tab
- `expect.poll()` is idiomatic Playwright for polling server endpoints like /health
