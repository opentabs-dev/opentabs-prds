## Codebase Patterns
- `getConfigDir()` reads `Bun.env.OPENTABS_CONFIG_DIR` lazily on each call (not cached) — safe for test isolation via env var override
- Bun test runner runs ALL test files in the same process — env var changes in one file affect others. Always re-assert env vars in `beforeEach` if other test files modify the same var
- Atomic file writes follow the pattern: write to `.tmp`, `chmod 0o600`, `rename` over target — used by `atomicWriteConfig` and `writeAuthFile`
- The managed extension lives at `~/.opentabs/extension/` — files written there are accessible to the Chrome extension via `chrome.runtime.getURL()`
- Extension offscreen document cannot access `chrome.storage` APIs — the background script relays storage values via `chrome.runtime.sendMessage`
- E2E tests that use the fixture's `mcpServer` have a secret configured (auto-generated UUID) — any endpoint that requires auth needs the secret passed through the helper chain
- CLI commands that query the server follow the pattern: `readConfig` → extract `config.secret` → set `Authorization: Bearer <secret>` header (see `audit.ts`, `status.ts`)
- When adding new fields to `OpentabsConfig`, update: `parseConfigRecord`, default config in `loadConfig`, `saveToolConfig` read-modify-write, and all test `OpentabsConfig` literals
- CLI flags/env vars that are determined once at startup follow the `dev-mode.ts` pattern: module-level `const`, exported via `isFoo()` function (see `skip-confirmation.ts`)
- E2E tests that create fake WebSocket connections while the real extension is connected must include the `connectionToken` from the authenticated `/health` endpoint — use `fetchConnectionToken(port, secret)` helper from fixtures.ts
- When adding new browser tool tests, set `state.skipConfirmation = true` unless specifically testing the permission flow — the permission engine defaults to 'ask' for unrecognized tools, which requires an extension connection
- The confirmation protocol uses fire-and-forget notifications (not request/response) for the server→extension direction, and the extension→server response resolves a stored Promise via `pendingConfirmations` Map
- Untracked `validate-icon.ts` / `validate-icon.test.ts` in `platform/plugin-tools/src/` are leftovers from an incomplete icon feature — delete them if present (they break the build)
- Non-configurable property descriptors (`configurable:false`) require rebuilding the parent object to remove entries — use `Object.getOwnPropertyDescriptor` loop to preserve other entries
- Session permissions from "Allow Always" are checked before config-based permissions — this order ensures user decisions during a session override the config file rules
- Side panel receives `confirmation.request` via `sp:serverMessage` (in `SIDE_PANEL_METHODS`) — handled in `handleNotification`, NOT as a separate message type
- Chrome API mocks in `message-router.test.ts` must include ALL APIs used at module top level — `action`, `notifications`, `sidePanel`, `windows.getCurrent` are all required
- React 19 strict mode: `Date.now()` in render triggers purity lint error — track timestamps in event handlers and pass as props

## 2026-02-22 - US-014
- Implemented: Add npm package signature verification for official plugins
- Files changed:
  - `platform/mcp-server/src/verify-plugin.ts` — New module: `verifyOfficialPlugin(packagePath, npmPackageName)` reads installed `_integrity`/`_shasum` from package.json, fetches npm registry dist metadata for the same version, compares hashes; `readInstalledIntegrity()` and `fetchRegistryDist()` are exported for unit testing
  - `platform/mcp-server/src/skip-verification.ts` — New module following `dev-mode.ts` pattern: `isSkipPluginVerification()` reads `--skip-plugin-verification` CLI flag and `OPENTABS_SKIP_PLUGIN_VERIFICATION` env var
  - `platform/mcp-server/src/discovery.ts` — Added Phase 3.5 verification step between loading and merging: `verifyOfficialPlugins()` runs verification in parallel for all official-tier npm plugins, downgrades failed plugins from 'official' to 'community' tier; skipped when `isSkipPluginVerification()` is true
  - `platform/mcp-server/src/verify-plugin.test.ts` — 11 unit tests covering: readInstalledIntegrity (reads hash fields, handles missing package.json, handles missing hash fields), fetchRegistryDist (returns dist data, handles 404, handles network error, handles missing dist field), verifyOfficialPlugin (integrity match, integrity mismatch, shasum fallback, shasum mismatch, registry unreachable, no installed hashes, no version, hash format mismatch)
- **Learnings for future iterations:**
  - The npm registry API `GET https://registry.npmjs.org/<package>/<version>` returns `{ dist: { integrity, shasum } }` — integrity is SRI format (sha512-...), shasum is hex SHA-1
  - Installed packages have `_integrity` and `_shasum` fields in their package.json (set by npm/bun during install) — not all installs include both fields
  - Verification is integrated into discovery.ts between Phase 3 (load) and Phase 4 (merge) — this ensures verification happens exactly once per discovery cycle
  - The `LoadedPlugin.trustTier` is readonly but the object is not frozen at this point in the pipeline, so `(plugin as { trustTier: TrustTier }).trustTier = 'community'` is safe for the downgrade
  - Network failures return `{ verified: true }` with a reason — this ensures air-gapped environments don't reject all official plugins
  - The 10-second fetch timeout uses `AbortController` for proper request cancellation

## 2026-02-22 - US-016
- Implemented: Add tool output sanitization with --dangerously-skip-sanitization bypass
- Files changed:
  - `platform/mcp-server/src/sanitize-tool-output.ts` — New module: `sanitizeToolOutput()` recursively walks object trees and applies path/URL/IP/localhost regex replacements to all string values (same patterns as `sanitizeErrorMessage`), leaves numbers/booleans/nulls/keys untouched, max depth 50
  - `platform/mcp-server/src/skip-sanitization.ts` — New module following `dev-mode.ts` pattern: `isCliSkipSanitization()` reads `--dangerously-skip-sanitization` CLI flag and `OPENTABS_SKIP_SANITIZATION` env var
  - `platform/mcp-server/src/config.ts` — Added optional `skipSanitization?: boolean` field to `OpentabsConfig`, parsing in `parseConfigRecord`, preservation in `saveToolConfig` read-modify-write
  - `platform/mcp-server/src/state.ts` — Added `skipSanitization: boolean` field to `ServerState`, defaults to `false` in `createState()`
  - `platform/mcp-server/src/reload.ts` — Propagates `skipSanitization` from config + CLI flag to state during reload (same pattern as `skipConfirmation`)
  - `platform/mcp-server/src/mcp-setup.ts` — Wired `sanitizeToolOutput` into both browser tool success path (line ~494) and plugin tool success path (line ~646); sanitization runs after `sanitizeOutput` (prototype pollution) but only when `state.skipSanitization` is false
  - `platform/mcp-server/src/http-routes.ts` — Added `outputSanitizationEnabled: !state.skipSanitization` to authenticated `/health` response
  - `platform/mcp-server/src/http-routes.test.ts` — Added `outputSanitizationEnabled` to `HealthResponse` interface
  - `platform/mcp-server/src/state.test.ts` — Added assertion for `skipSanitization` default value
  - `platform/mcp-server/src/sanitize-tool-output.test.ts` — 12 unit tests covering: Unix paths, Windows paths, URLs, localhost, IPs, combined patterns, recursive objects, recursive arrays, key preservation, primitives, long strings, deep nesting
- **Learnings for future iterations:**
  - The sanitize regex order matters: Unix path regex (`/foo/bar`) runs before URL regex (`https://...`), so URLs containing paths get the path portion replaced first, resulting in `http[PATH]` rather than `[URL]` — this matches `sanitizeErrorMessage` behavior
  - The `sanitizeToolOutput` function does NOT truncate (unlike `sanitizeErrorMessage` which caps at 500 chars) — tool output can be arbitrarily long and truncation would corrupt data
  - The bypass flag follows the triple-source pattern: CLI flag (`--dangerously-skip-sanitization`) OR env var (`OPENTABS_SKIP_SANITIZATION=1`) OR config.json (`skipSanitization: true`) — any one activates the bypass
  - Error sanitization (via `sanitizeErrorMessage`) is deliberately NOT affected by the `skipSanitization` flag — errors are always sanitized regardless
  - The `sanitizeToolOutput` runs AFTER `sanitizeOutput` (prototype pollution prevention) — order matters because `sanitizeOutput` restructures the object while `sanitizeToolOutput` only transforms string values
