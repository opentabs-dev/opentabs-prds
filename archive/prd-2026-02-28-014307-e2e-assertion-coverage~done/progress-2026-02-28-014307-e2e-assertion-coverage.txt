## Codebase Patterns
- E2E test assertions: use `expect(array).toContainEqual(expect.objectContaining({...}))` instead of `find()+toBeDefined()` or `Set.has().toBe(true)` ‚Äî shows actual array on failure
- `expect.stringContaining('...')` works inside `objectContaining` for partial string matches in toContainEqual
- `expect.stringMatching(/regex/)` also works inside `objectContaining` for regex matches
- E2E tests use Phase 1 (no test:e2e) for stories with `e2eCheckpoint: false`
- Browser tools E2E tests use `testServer` fixture; to add WebSocket support, add a `/ws` upgrade handler and `/ws-test` HTML page to `e2e/test-server.ts`
- The `ws` package is a root dependency ‚Äî available for import in E2E test server code
- Import order: `ws` (third-party) must come before `node:http` (node builtins) per ESLint import-x/order rule
- `waitForWebSocketFrames` polling helper follows the same pattern as `waitForNetworkRequests` in browser-tools.e2e.ts
- Side panel detection: `extension_get_side_panel` sends `chrome.runtime.sendMessage({ type: 'sp:getState' })` from the background; App.tsx listens and responds with `{ state, html }`; timeout (3s) ‚Üí `{ open: false }`
- Opening side panel via Playwright (`openSidePanel(extensionContext)`) works with the detection mechanism ‚Äî the React app registers the listener and responds to `sp:getState`
- Use `waitFor` polling after `openSidePanel` to wait for React to mount before asserting `open: true`
