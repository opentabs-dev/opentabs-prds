## Codebase Patterns
- OpentabsConfig interface is in `platform/mcp-server/src/config.ts`, exported as a type
- Config JSON key for permissions is now `permissions` (was `plugins`)
- `parseConfigRecord` in config.ts normalizes raw JSON to OpentabsConfig
- `savePluginPermissions` does a read-modify-write: reads config from disk, replaces the permissions field, writes back
- Test fixtures in reload.test.ts write config.json directly via `writeFileSync` with JSON fixtures â€” these must match the config schema
- `state.registry.plugins` and `state.pluginPermissions` are different things: registry.plugins is the plugin registry (Map), pluginPermissions is the permission config from config.json
- The `isLocalPathEntry` helper was only used by legacy migration code and is now removed
- The CLI config command (`platform/cli/src/commands/config.ts`) has its own raw `Record<string, unknown>` manipulation for `config.permissions` (via `ensurePluginEntry`, `pruneEmptyPluginEntries`, `normalizeConfigForDisplay`) â€” these are separate from the mcp-server's typed OpentabsConfig helpers
- E2E `OpentabsConfig` in `e2e/fixtures.ts` is a separate type from the server's â€” must be kept in sync manually when schema changes
- `HealthResponse.plugins` (and reload response `plugins: number`) is the loaded plugin count, NOT the config key â€” never rename it during config schema changes
