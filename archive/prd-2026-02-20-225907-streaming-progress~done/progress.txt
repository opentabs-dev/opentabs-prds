## Codebase Patterns
- ToolDefinition interface is in platform/plugin-sdk/src/index.ts ‚Äî the single source of truth for tool types
- The IIFE wrapper in platform/plugin-tools/src/commands/build.ts wraps tool.handle() with lifecycle hooks (onToolInvocationStart/End) ‚Äî any changes to handle()'s signature must also be reflected in the wrapper's origHandle.call() forwarding
- Tool execution happens in tool-dispatch.ts via chrome.scripting.executeScript in MAIN world ‚Äî the injected function calls tool.handle(tInput, context) with a ToolHandlerContext
- Tests in plugin-sdk use Promise.resolve() instead of async/await for simple handlers to avoid the @typescript-eslint/require-await lint rule
- Prettier enforces no parens around single arrow function parameters
- CustomEvents fired in MAIN world ARE visible to ISOLATED world content scripts ‚Äî they share the same DOM. This is the correct cross-world communication pattern for Chrome extensions.
- The dispatchId is the JSON-RPC request ID (UUID string) ‚Äî it correlates progress notifications from adapter ‚Üí extension ‚Üí server back to the pending dispatch
- InternalMessage is a discriminated union in types.ts ‚Äî adding a new message type requires updating exhaustiveness checks in background.ts AND offscreen/index.ts switch statements
- McpServerInstance.setRequestHandler passes (request, extra) where extra has signal, _meta (with progressToken), sessionId, and sendNotification ‚Äî test mocks must match this signature
- dispatchToExtension injects dispatchId into the params sent to the extension so the extension can use it for progress correlation
- The extension-side timeout in tool-dispatch.ts uses a progressCallbacks Map keyed by dispatchId ‚Äî background.ts calls notifyDispatchProgress() to reset the timeout when progress arrives
- ESLint rule import-x/exports-last requires all exports at the end of file ‚Äî use `export { a, b }` instead of inline `export const`
- MAX_SCRIPT_TIMEOUT_MS (extension) is 295s = MAX_DISPATCH_TIMEOUT_MS (server 300s) minus 5s safety margin, ensuring the extension always responds before the server gives up
- **CRITICAL**: Bun's bundler strips unused function parameters during compilation. The lifecycle hook wrapper in build.ts must use `...args` rest params with `origHandle.apply(this, args)` instead of named params like `(params, context)` + `origHandle.call(this, params, context)`. Bun compiles the latter to just `(params)` + `origHandle.call(this, params)`, silently dropping the context parameter.
- The MCP SDK's SSE transport sends progress notifications as inline events in the same SSE stream as the tool response. The client receives `event: message\ndata: {method: "notifications/progress", ...}` events before the final `data: {result: ..., id: N}` event. To capture progress, parse ALL SSE data lines and filter by `method === 'notifications/progress'`.
- The `_meta.progressToken` field must be sent in the tools/call request's `params` object for the MCP server to emit progress notifications. Without it, `onProgress` is undefined and progress is silently dropped.
