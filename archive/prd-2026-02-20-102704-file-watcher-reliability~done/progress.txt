## Codebase Patterns
- FileWatcherEntry is defined in state.ts and used in file-watcher.ts â€” all fields must be initialized in both watchPlugin and watchPendingPlugin return values
- statSync with `{ throwIfNoEntry: false }` is the safe way to check file mtime without throwing on missing files
- File watcher handlers (handleIifeChange, handleManifestChange, handlePendingPluginChange) receive pluginDir but need to look up their FileWatcherEntry from state.fileWatcherEntries by matching pluginDir
- The config watcher's debounce callback captures variables from the enclosing scope (configDir, gen) â€” any new variables (like configPath) must also be declared in that outer scope to be accessible inside the callback
- Pending plugins are identified by pluginName starting with `(pending:` â€” use this convention to distinguish loaded vs pending entries
- All timer/interval IDs stored on ServerState must be cleaned up in stopFileWatching for hot-reload safety
- Polling functions should capture fileWatcherGeneration at creation time and self-clear if it changes

- Added `mtimePollTimerId: ReturnType<typeof setInterval> | null` to ServerState (default null)
- Created `startMtimePolling(state, callbacks)` function that polls every 30s:
  - Stats all watched files (manifest + IIFE per plugin entry) and config.json
  - Compares mtime to `lastSeenMtimes`; if newer, calls the appropriate handler

  - Captures fileWatcherGeneration and self-terminates if generation changes (hot reload)
- Called `startMtimePolling` from `startFileWatching` after all fs.watch watchers are set up

- Files changed: platform/mcp-server/src/state.ts, platform/mcp-server/src/file-watcher.ts
- **Learnings for future iterations:**
