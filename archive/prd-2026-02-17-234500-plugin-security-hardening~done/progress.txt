## Codebase Patterns
- Manifest schema validation lives in `platform/mcp-server/src/manifest-schema.ts` using Zod; add length limits via `.max()` on string fields
- `checkBrowserToolReferences` in `discovery.ts` is an exported pure function for detecting browser tool name references in tool descriptions — testable without mocking
- ESLint `import-x/exports-last` rule requires all exports at the bottom of the file; use non-export `const` and a single `export { ... }` at the end
- URL pattern validation lives in `platform/shared/src/index.ts` in `validateUrlPattern`
- IIFE wrapper template is in `platform/cli/src/commands/build.ts:bundleIIFE` — the wrapper runs at injection time in MAIN world
- The hash setter snippet is appended AFTER the main IIFE (not inside it) — code in the appended snippet runs after the adapter is registered
- Use `writable:false, configurable:true` for frozen adapter properties — this prevents page scripts from overwriting via assignment while allowing re-injection (Object.defineProperty) and cleanup (Reflect.deleteProperty)
- The IIFE wrapper must `Reflect.deleteProperty` before re-assigning the adapter entry — otherwise the non-writable property descriptor silently rejects the assignment in sloppy mode
- Plugins in `plugins/` use the published npm CLI, not the local one — CLI changes only affect plugins after publishing
- Validation tests are in `platform/shared/src/validation.test.ts`
- The validation function returns `string | null` — null means valid, string is the error message
- URL pattern parsing extracts scheme, host, and path via regex: `/^(\*|https?):\/\/(.+?)(\/.*)$/`
- Host validation allows: `*`, `localhost(:port)?`, or `(*.)?domain.tld` patterns
- Config is loaded in `config.ts:loadConfig()` and persisted in `config.ts:saveConfig()`
- Config fields: `plugins` (local paths), `tools` (enabled/disabled map), `secret` (WS auth), `npmPlugins` (allowed npm packages)
- Plugin discovery call chain: `reload.ts:reloadCore` → `loadConfig()` → `discoverPlugins(paths, npmPlugins)` → parallel `discoverFromNodeModules` + `discoverFromLocalPaths`
- Config is persisted from `http-routes.ts:onToolConfigPersist` callback — must include all state fields
- `ServerState` in `state.ts` mirrors config fields: `pluginPaths`, `npmPlugins`, `toolConfig`, `wsSecret`
- E2E tests have their own `OpentabsConfig` interface in `e2e/fixtures.ts` — keep in sync with the main one
- MCP endpoint auth uses Bearer token from `state.wsSecret`; `/health` and `/ws-info` are unauthenticated
- `checkBearerAuth` in `http-routes.ts` returns null (pass) or Response(401); when wsSecret is null, auth is bypassed
- E2E `McpServer` interface has a `secret` field populated from config at startup; pass it to `createMcpClient`
- When writing test configs after server startup, always preserve the `secret` field — the server auto-generates secrets, so overwriting config without secret causes auth failures
- `loadConfig` fallback path (corrupt JSON) returns a new random secret that is NOT persisted to disk
- ESLint import order: relative (`./`) before external type imports; `bun run lint:fix` auto-sorts
- `trustTierPrefix` in `mcp-setup.ts` maps TrustTier → description prefix string; browser tools are NOT prefixed
- `ToolLookupEntry` in `state.ts` includes `validate` (pre-compiled Ajv function) and `validationErrors` (error formatter) — compiled at discovery time in `rebuildToolLookups`
- Ajv is used for JSON Schema validation of plugin tool inputs; validators are pre-compiled per tool for performance
- ESLint `import-x/no-named-as-default` rule prohibits using the same name as a named export for default imports — use a different identifier (e.g., `import AjvValidator from 'ajv'` instead of `import Ajv from 'ajv'`)
- ESLint `@typescript-eslint/no-non-null-assertion` prohibits `!` assertions in tests — use `if (!x) throw new Error(...)` for narrowing instead

## 2026-02-18 - US-006
- Implemented adapter entry freezing in the IIFE wrapper generated by `opentabs build`
- Modified the hash setter snippet (appended after the main IIFE) to also freeze the adapter:
  1. Sets `__adapterHash` on the adapter object
  2. Freezes each individual tool entry with `Object.freeze(tool)`
  3. Freezes the `tools` array with `Object.freeze(adapter.tools)`
  4. Freezes the adapter object itself with `Object.freeze(adapter)`
  5. Redefines the property on the adapters map with `Object.defineProperty` using `writable:false, configurable:true, enumerable:true`
- Added `Reflect.deleteProperty` in the IIFE wrapper's teardown flow before re-assigning the adapter entry — without this, the non-writable property descriptor silently rejects the new assignment in sloppy mode
- Protection model: `writable:false` prevents page scripts from overwriting via simple assignment; `configurable:true` allows re-injection (new IIFE deletes and re-creates the property) and extension cleanup (Reflect.deleteProperty)
- Files changed: `platform/cli/src/commands/build.ts`
- **Learnings for future iterations:**
  - The IIFE generated by `Bun.build({ format: 'iife' })` runs in sloppy mode — assignments to non-writable properties silently fail rather than throwing
  - The hash setter snippet runs as a separate self-invoking function appended after the main IIFE, making it the ideal place for post-registration locking
  - `Object.defineProperty` with `configurable:true` allows re-definition but prevents simple `obj[key] = value` assignment — a good balance between security and re-injectability
  - Plugins use the published CLI from npm, not the local workspace CLI — changes to the CLI only affect plugin builds after publishing
