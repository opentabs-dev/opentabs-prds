## Codebase Patterns
- getAllToolsList() always returns every tool (no filtering) with description prefixes based on permission state
- checkToolCallable() is a pure existence check; permission gating happens at dispatch time in handlePluginToolCall/handleBrowserToolCall
- ToolPermission type ('off' | 'ask' | 'auto') replaces TrustTier and enabled boolean — all tools default to 'off'
- WireToolDef, WirePluginPayload, ConfigStatePlugin, ConfigStateBrowserTool all use `permission: ToolPermission` field
- Config param types renamed: ConfigSetTool*Enabled*Params → ConfigSetTool*Permission*Params (enabled→permission throughout)
- OFFICIAL_SCOPE constant removed — use literal '@opentabs-dev' where the scope is needed
- RegisteredPlugin no longer has trustTier; use `source: 'npm' | 'local'` to distinguish plugin origin
- Extension message types renamed: BgSetToolEnabledMessage → BgSetToolPermissionMessage, bridge functions similarly renamed
- ConfirmationRequest/ConfirmationResponse are new shared types for the simplified confirmation flow
- serializePluginForExtension takes (state, plugin) and resolves permissions via getToolPermission()
- OpentabsConfig now uses `plugins: Record<string, PluginPermissionConfig>` — no more `tools`, `browserToolPolicy`, `permissions` fields
- savePluginPermissions() replaces saveToolConfig() and saveBrowserToolPolicy() — single read-modify-write
- state.pluginPermissions stores per-plugin permissions from config; state.toolConfig/browserToolPolicy/permissions have been removed
- PendingConfirmation.resolve accepts ConfirmationDecision {action: 'allow'|'deny', alwaysAllow: boolean}
- sendConfirmationRequest signature: (state, tool, plugin, params) → Promise<ConfirmationDecision>
- DispatchCallbacks interface has onToolConfigChanged, wired to server.sendToolListChanged in registerMcpHandlers
- McpCallbacks uses onPluginPermissionsPersist (replaces onToolConfigPersist + onBrowserToolPolicyPersist)
- handleConfigSetToolPermission is the unified handler for per-tool permissions (plugin tools + browser tools via plugin='browser')
- handleConfigSetPluginPermission is the unified handler for plugin-level defaults (plugins + browser via plugin='browser')
- Extension protocol method names: config.setToolPermission, config.setPluginPermission
- Both handleBrowserToolCall and handlePluginToolCall accept DispatchCallbacks as last parameter
- Browser tools use 'browser' as pluginName for permission resolution and confirmation requests
- Plugin tools pass raw toolBaseName (unprefixed) to getToolPermission and sendConfirmationRequest
- Browser pseudo-plugin permission is transmitted as ConfigStateResult.browserPermission (separate from plugins array)
- pendingPluginPermissionUpdates in server-state-cache handles both plugin entries and browser pseudo-plugin (cache.browserPermission)
- skipPermissions flows end-to-end: server state → ConfigStateResult → sync.full/plugins.changed → ServerStateCache → handleBgGetFullState → App.tsx → child components; when adding new fields to ConfigStateResult, update ServerStateCache + EMPTY_CACHE + loadServerStateCacheFromSession + sync.full/plugins.changed handlers in message-router.ts + handleBgGetFullState response
