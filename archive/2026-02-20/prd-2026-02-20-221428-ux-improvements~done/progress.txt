## Codebase Patterns
- Config type `OpentabsConfig` is in `platform/mcp-server/src/config.ts` — the MCP server's config module
- CLI has its own config helpers in `platform/cli/src/config.ts` — separate from the MCP server's config
- The CLI helper `getLocalPluginsFromConfig` reads raw JSON, while the MCP server's `parseConfigRecord` does full validation
- `config.plugins` in `state.registry.plugins` refers to the Map of RegisteredPlugin objects, NOT the config field — don't confuse them
- Test files that use CLI commands via `runCli()` read config via `readTestConfig()` which returns raw JSON — field names must match what the CLI writes
- Doctor tests call `checkPlugins()` directly — the function reads `localPlugins` from the raw config object passed in
- The `create-plugin` package auto-registers scaffolded plugins in `localPlugins` via `scaffold.ts`'s `autoRegisterPlugin`
- `Bun.spawnSync` has overloaded signatures — mocking it requires casting through a spread+unknown pattern rather than direct `mock((args: string[]) => ...)` which doesn't match the overload types
- JSDoc comments with backtick-quoted text containing `*/` (e.g., `` `@*/opentabs-plugin-*` ``) will prematurely terminate the comment block — avoid `*/` inside backtick-quoted text in JSDoc
- `RegisteredPlugin` and `LoadedPlugin` both have a `source: PluginSource` field ('npm' | 'local') — all test factory functions that construct these types need to include `source: 'local' as const`
- When adding a required field to a widely-used interface (e.g., `RegisteredPlugin`), search for all `makePlugin`/`createPlugin` test helpers across the entire test suite — there are multiple independent ones scoped within different `describe` blocks
- `plugin-tools` is standalone — cannot import from CLI or MCP server packages. Config helpers must be duplicated as lightweight versions in `build.ts`
- `OPENTABS_CONFIG_DIR` env var is used across all packages (MCP server, CLI, plugin-tools) for test isolation of config paths
- The `import-x/exports-last` rule requires all exports at the end of the file — use `export { ... }` blocks, not inline `export const/interface`
- `installExtension()` in `setup.ts` is reusable — takes configDir as parameter, returns `InstallExtensionResult` with installed/firstTime/extensionDest/version
