## Codebase Patterns
- The `OpenTabsPlugin` class uses optional method syntax (`method?(): void`) for hooks that plugins may or may not implement
- Existing pattern: `teardown?()` — new lifecycle hooks follow the same style
- SDK interface changes require publishing via `./scripts/publish.sh` before plugins can consume them
- Plugin SDK source: `platform/plugin-sdk/src/index.ts`
- The IIFE wrapper in `bundleIIFE()` is a TypeScript template string bundled by `Bun.build` as an IIFE targeting the browser
- Tool mutations (handle wrapping) must happen in the IIFE wrapper BEFORE `hashAndFreeze` which appends `Object.freeze` calls
- To prevent double-fire of `onDeactivate`, the pattern is: capture in closure, set `plugin.onDeactivate = undefined`, and call via the wrapped `teardown`

## 2026-02-21 08:42 — US-004
- Updated `bundleIIFE` wrapper code in `platform/plugin-tools/src/commands/build.ts` to wire all lifecycle hooks
- **onActivate**: called after `adapters[name] = plugin` registration, wrapped in try-catch
- **onDeactivate**: called on the OLD adapter before teardown during re-injection; for the NEW adapter, `onDeactivate` is captured in closure and fired via a wrapped `teardown` function (then set to `undefined` on the plugin to prevent double-fire when the frozen adapter is later torn down)
- **onNavigate**: intercepts `history.pushState`/`history.replaceState` and adds `popstate`/`hashchange` listeners; only if `plugin.onNavigate` exists at runtime (no overhead otherwise); cleanup restores original history methods and removes event listeners
- **onToolInvocationStart/End**: wraps each `tool.handle()` with `performance.now()` timing before `hashAndFreeze` freezes the tools; only if either hook exists on the plugin
- All hook calls wrapped in try-catch to prevent errors from breaking tool execution or adapter registration
- Files changed: `platform/plugin-tools/src/commands/build.ts`
- **Learnings for future iterations:**
  - The IIFE wrapper code is a TypeScript template string in `bundleIIFE()` — it's bundled by `Bun.build` as an IIFE targeting the browser
  - Tool wrapping must happen BEFORE the `hashAndFreeze` snippet (appended after bundling) which calls `Object.freeze` on each tool
  - The teardown wrapping pattern: capture `origTeardown` and `origOnDeactivate` in closure, assign a new wrapper to `plugin.teardown`, set `plugin.onDeactivate = undefined` to prevent double-fire on re-injection (since the wrapper already calls `origOnDeactivate`)
  - The `hashAndFreeze` snippet runs after the IIFE, so mutations done in the wrapper (tool.handle wrapping, teardown wrapping) are preserved before freeze
  - Navigation listener cleanup (restore `history.pushState`/`replaceState`, remove event listeners) is done inside the wrapped `teardown`, ensuring clean state on adapter replacement
- The e2e-test plugin's npm-installed `@opentabs-dev/plugin-tools` must include lifecycle wiring code from US-004. Before publishing, use `bun link` to link local platform packages into the plugin for testing.
- E2E tests use `waitFor()` polling instead of fixed timeouts to check globalThis markers — more reliable and faster
- The `setupToolTest()` helper waits for adapter injection + tab ready state, so hook markers like `__opentabs_onActivate_called` are available immediately after
- Hot reload (`mcpServer.triggerHotReload()`) causes sync.full → re-injection → teardown of old adapter (fires onDeactivate) → new adapter registered (fires onActivate)
