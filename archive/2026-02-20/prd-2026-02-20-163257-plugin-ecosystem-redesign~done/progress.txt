## Codebase Patterns
- Shared types/utilities live in `platform/shared/src/` and are re-exported from `index.ts`
- The shared package uses `.js` extensions in imports (ESM convention with TypeScript)
- Prettier enforces single-line exports when they fit, and parenthesized ternaries for multi-line arrow returns
- Pre-commit hooks run prettier, eslint, and knip — formatting must be correct before committing
- ESLint `import-x/exports-last` requires all exports at the bottom of the file — declare without `export`, then export at the end
- ESLint `import-x/order` requires value imports before type imports from the same module
- ESLint `import-x/order` within node: protocol is alphabetical (e.g., `node:fs/promises` before `node:os`)
- In barrel files, use separate `export { ... }` and `export type { ... }` lines rather than mixed type/value exports
- New modules must be reachable from existing import chains or knip will flag them as unused files
- When accessing `unknown[]` elements, assign to `unknown` before casting to `Record<string, unknown>` — avoids `@typescript-eslint/no-unnecessary-condition` on typeof checks
- Knip entry `src/**/*.test.ts` makes test files valid entry points — new modules referenced only from tests are reachable
- When rewriting a module with active consumers, use a `-legacy.ts` sibling to preserve old code until all consumers are migrated
- File watcher watches `dist/tools.json` and `dist/adapter.iife.js` in the `dist/` directory (not the plugin root)
- `discovery-legacy.ts` and `manifest-schema.ts` have been deleted — all discovery goes through loader.ts
- E2E test `expect(logsJoined).not.toContain('error')` is too broad — use specific error patterns to avoid matching "0 error(s)" in informational log messages
- Test helpers creating plugins must use new format: `package.json` (with `opentabs` field) + `dist/tools.json` + `dist/adapter.iife.js`
