## Codebase Patterns
- Plugin SDK types and factories are defined directly in `platform/plugin-sdk/src/index.ts` (not in separate files)
- Factory functions (`defineTool`, `defineResource`, `definePrompt`) are identity functions that provide type inference — no runtime validation
- The `OpenTabsPlugin` base class uses `abstract readonly` for required properties and plain `readonly` with `?` for optional ones
- Test file for SDK exports is `platform/plugin-sdk/src/index.test.ts` — tests cover type compilation, factory identity, and runtime behavior
- Commit messages follow `feat: US-XXX — description` pattern
- `tools.json` format changed from plain array to `{ tools: [...], resources: [...], prompts: [...] }` — loader and file-watcher support BOTH formats for backward compatibility
- Manifest types (`ManifestResource`, `ManifestPrompt`, `ManifestPromptArgument`) are in `@opentabs-dev/shared` — build.ts imports from there
- `RegisteredPlugin` and `LoadedPlugin` include `resources: ManifestResource[]` and `prompts: ManifestPrompt[]` — all test helpers must include these fields
- The IIFE adapter already exposes `resources` and `prompts` via `adapters[name] = plugin` — no IIFE wrapper changes needed
- The `makePlugin()` test stub doesn't have real Zod schemas, so `generateManifest` tests must use `makeRealTool()` instead
- `PluginRegistry` has O(1) lookup maps: `toolLookup`, `resourceLookup`, `promptLookup` — all populated during `buildRegistry()`
- Resource URI prefixing: `opentabs+<plugin>://<originalUri>`; prompt name prefixing: `<plugin>_<promptName>` (same as tools)
- MCP SDK exports `McpError` and `ErrorCode` from `@modelcontextprotocol/sdk/types.js` — use for handler errors
- `McpServerInstance` interface has notification methods: `sendToolListChanged`, `sendResourceListChanged`, `sendPromptListChanged`
- All mock McpServerInstance objects in test files must be kept in sync when methods are added to the interface
- When adding new WebSocket dispatch methods: update `DISPATCH_METHODS` in `known-methods.ts`, add handler to `methodHandlers` in `message-router.ts`, mock in both `known-methods.test.ts` and `message-router.test.ts`
- After changing server log messages, search E2E tests for regex patterns matching the old format — E2E tests assert on log content
- Extension wraps ALL dispatch results in `{ output: ... }` — dispatch handlers must unwrap: `const raw = result as Record<string, unknown>; const data = raw.output ?? raw;`
- When tools.json format changes, E2E tests in `hot-reload-dynamic.e2e.ts` that read/write tools.json directly must be updated — use format-aware helpers
- Publish workflow: `./scripts/publish.sh <version>` handles version bump, cross-references, rebuild, and publish in dependency order
- When switching plugins from `file:` back to npm, delete `node_modules` and `bun.lock` first for a clean resolve
