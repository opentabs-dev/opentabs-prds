## Codebase Patterns
- Plugin SDK utility modules follow: types/interfaces at top (non-exported), implementation in middle, exports at bottom (`import-x/exports-last` rule)
- ESLint import order: value imports from local modules first, then external (bun:test, etc.), then type-only imports
- Use optional chaining (`?.`) instead of non-null assertions (`!`) in test files — `@typescript-eslint/no-non-null-assertion` is enforced
- `ignoreExportsUsedInFile: true` in knip config — internal exports used within the same file are fine
- Use `func-style` as arrow expressions (`const fn = () => ...`) not function declarations
- `@typescript-eslint/restrict-template-expressions` enforced — template literal interpolations must be string-typed (use `?? ''` for `string | undefined`)
- `@typescript-eslint/no-base-to-string` enforced — don't use `String(value)` on objects, use targeted serialization
- Avoid `(v as bigint)` casts when TypeScript already narrows the type via `typeof` checks
- Extension has NO content scripts — uses `chrome.scripting.executeScript` from background to interact with tabs in both MAIN and ISOLATED worlds
- When IIFE wrapper needs SDK internals, use `globalThis.__openTabs` registration instead of direct imports (plugins may have older SDK versions installed from npm)
- New InternalMessage types must be added to exhaustiveness switch in both `background.ts` AND `offscreen/index.ts`
- Use `window as unknown as Record<string, unknown>` (not `window as Record<string, unknown>`) in ISOLATED world scripts for TypeScript type safety
- `McpServerInstance` in `mcp-setup.ts` is a hand-typed interface subset — adding methods requires updating ALL mock objects in test files (extension-protocol, http-routes, mcp-setup, mcp-setup-dispatch, reload tests)
- `McpCallbacks` bridges extension-protocol to MCP sessions — new callbacks need `createMcpCallbacks` in `http-routes.ts` + all test mocks
- E2E tests verify logging via MCP server stdout logs (`mcpServer.logs`) and `/health` endpoint `logBufferSize` — the streamable HTTP MCP client doesn't support async notifications
- When e2e-test plugin needs unreleased SDK features, use `bun link` to link local platform packages (`@opentabs-dev/plugin-sdk`, `@opentabs-dev/plugin-tools`) — then rebuild the plugin
- Register new SDK runtime objects on `globalThis.__openTabs` (e.g., `ot.log = log`) so plugins can access them via globalThis when the npm SDK version doesn't export them yet
- `HealthResponse` in `e2e/fixtures.ts` must be extended when new fields are added to the `/health` endpoint (e.g., `pluginDetails` with `logBufferSize`)
