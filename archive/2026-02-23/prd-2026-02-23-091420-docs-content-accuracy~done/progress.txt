## Codebase Patterns
- Docs quality checks: `cd docs && bun run build && bun run type-check && bun run lint && bun run knip`
- Docs dependencies must be installed separately (`bun install` in docs/) since docs is standalone
- The root pre-commit hook runs root-level knip which covers platform/ packages — fixing root knip issues may be needed even for docs-only changes
- CLI source of truth: platform/cli/src/commands/ contains the actual flag definitions and behavior
- The docs project uses MDX with JSX table components (`<Table>`, `<Table.Row>`, etc.)
- The CliCommand component auto-derives bun/pnpm/yarn commands from `npmCommand` prop — no need to duplicate
- GitHub org is `opentabs-dev`, not `AnomalyCo`
- Scaffold creates directory with raw name (`my-app`), npm package is `opentabs-plugin-my-app`
- Publish script is TypeScript: `bun scripts/publish.ts <version>` — publishes 5 packages in order
- SDK source of truth for types: platform/plugin-sdk/src/index.ts (ToolDefinition, ProgressOptions), errors.ts (ToolError), dom.ts, fetch.ts
- ProgressOptions fields `progress` and `total` are optional (for indeterminate progress)
- ToolDefinition `displayName` and `icon` are optional (auto-derived/defaulted at build time)
- All 6 ToolError factory methods accept optional `code` parameter: auth(msg, code?), notFound(msg, code?), rateLimited(msg, retryAfterMs?, code?), validation(msg, code?), timeout(msg, code?), internal(msg, code?)
- fetchFromPage timeout uses `ToolError.timeout()` → code is `'TIMEOUT'` (uppercase), not `'timeout'`
- fetchJSON parse failure uses `ToolError.validation()` → code is `'VALIDATION_ERROR'`, not `'json_parse_error'`
- waitForSelector has generic type parameter `<T extends Element = Element>`
- All fetch JSON utilities support Zod schema as last arg for type-safe response validation (fetchJSON, postJSON, putJSON, patchJSON, deleteJSON, postForm, postFormData)
- PromptDefinition.args (Zod schema) auto-generates arguments metadata during build and types the render() function
- ResourceDefinition.schema is type-level only (no runtime validation); use ResourceContentType<> to derive content types
