## Codebase Patterns
- Import order in browser-extension src: local modules alphabetically, then @opentabs-dev/shared, then type imports
- Prettier enforces no parens on single-param arrow functions (e.g., `tabId => ...` not `(tabId) => ...`)
- The ESLint config's `strictTypeChecked` applies globally; non-TS files (like .mjs) need to be in global ignores
- `mock.module` in bun:test only mocks the module path used at import-time — export signatures must match exactly
- When mocking modules for dispatch-helpers tests, mock ALL exports from each module (not just the ones under test) to avoid import errors
- Use `import type` for `DispatchResult` from `dispatch-helpers.js` — it's the shared type for executeOnTab callback return values in tests
- Avoid non-null assertions (`!`) in test files — ESLint's `strictTypeChecked` forbids them. Use a helper function with `expect().toBeGreaterThanOrEqual()` + explicit null check instead
- The `DispatchResult` type is now the canonical shared type in `dispatch-helpers.ts` for all three dispatch paths (tool, resource, prompt)
- When splitting files into subdirectories, sibling imports (`./`) must come before parent imports (`../`) in import-x/order
- The tsconfig.json in browser-extension includes `src/**/*.ts` so new files in subdirectories are automatically covered
- The `withDebugger` helper is exported from `resource-commands.ts` and imported by `interaction-commands.ts` for `handleBrowserHandleDialog`
- The `isCapturing` function from `network-capture.ts` is only needed by `resource-commands.ts` (for `withDebugger`), not by `network-commands.ts` which uses `startCapture`/`stopCapture`/`getRequests`/etc.
