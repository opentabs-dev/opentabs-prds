## Codebase Patterns
- JSON-RPC error codes are defined as named constants in `platform/browser-extension/src/json-rpc-errors.ts`
- Error codes inside `chrome.scripting.executeScript` `func` callbacks MUST remain as raw numbers because those functions run in the page's MAIN world and cannot import extension modules
- Non-error constants (timeouts, limits, keys) CAN be passed to executeScript funcs via the `args` array — the constant is defined once in constants.ts and passed through at call time
- The browser extension uses `sendToServer` from `messaging.ts` for all JSON-RPC responses to the MCP server
- `sendTabStateNotification` in `messaging.ts` is the single place for tab.stateChanged payloads to both server and side panel
- `PluginTabStateInfo` interface in `extension-messages.ts` is the shared type for tab state info `{ state, tabId, url }`
- `sendTabSyncAll` in tab-state.ts has a different shape (batched tab.syncAll to server, individual tab.stateChanged to side panel only) — it does NOT use sendTabStateNotification
- Import order in source files is enforced by `import-x/order` ESLint rule — imports must be alphabetical within each group
- After editing code with longer constant names, run `bun run lint:fix` to auto-fix line-length formatting issues
- TypeScript narrows `Record<string, unknown>` values after `typeof` checks — do NOT add `as number` casts after `typeof x === 'number'` guards, as ESLint flags them as unnecessary
- Confirmation badge/notification logic lives in `confirmation-badge.ts` — the chrome.notifications.onClicked listener is registered via `initConfirmationBadge()` called from background.ts (not a module-level side effect)
- When extracting a new module from an existing one, all test files that mock the original module may need to also mock the new module
- message-router.ts has three wrapper patterns: `wrapAsync` (async + id guard), `wrapSync` (sync + id guard + try-catch), `wrapNotification` (async, no id guard — for notifications). `handleExtensionReload` is a special case assigned directly as MessageHandler.
- `background-message-handlers.ts` uses a `Map<InternalMessage['type'], MessageHandler>` dispatch map for chrome.runtime.onMessage routing — handlers are `Record<string, unknown>` typed for runtime validation
- Side panel toggle logic lives in `side-panel-toggle.ts` — initialized via `initSidePanelToggle()` from background.ts
- `buildWsUrl(port)` in constants.ts is the single source of truth for the `ws://localhost:${port}/ws` URL pattern
- The `import-x/exports-last` ESLint rule requires all exports at the bottom of the file — use a single `export { ... }` block
- `queryMatchingTabIds(urlPatterns)` in iife-injection.ts is the shared helper for tab collection/deduplication — both `injectPluginIntoMatchingTabs` and `cleanupAdaptersInMatchingTabs` use it
- `teardownAdapterInTab(tabId, pluginName)` in iife-injection.ts wraps the simple adapter teardown executeScript call
- `fetchWsInfo(httpBase)` in offscreen/index.ts encapsulates the /ws-info fetch + 401 retry pattern — returns `{ response }` or `{ reason }` discriminated union
- `wsToHttpBase(wsUrl)` in offscreen/index.ts converts ws:// URLs to http:// base URLs
- `disconnectAndReconnect(closeReason)` in offscreen/index.ts handles WebSocket close + cancel pending timer + immediate reconnect
- `DisconnectReason` type has three variants: `'connection_refused'` (server unreachable), `'auth_failed'` (401), `'timeout'` (pong watchdog)
- `requireStringParam(params, paramName, id)` in helpers.ts validates a non-empty string param — follows the same pattern as `requireTabId`/`requireSelector` (returns value or null, sends JSON-RPC error as side effect)
- `sendValidationError(id, message)` in helpers.ts sends JSONRPC_INVALID_PARAMS (-32602) errors — use instead of raw `sendToServer` for parameter validation and "not found" errors
- `sendErrorResult(id, err)` in helpers.ts sends JSONRPC_INTERNAL_ERROR (-32603) — accepts Error objects for custom messages
- No browser command handler file should import `sendToServer` directly — all responses go through `sendSuccessResult`, `sendErrorResult`, or `sendValidationError`
- E2E fixtures patch `DEFAULT_SERVER_PORT` in bundled offscreen.js (not a literal URL) — resilient to bundler output changes
- `handleBrowserPressKey` lives in `key-press-command.ts`, `handleBrowserScroll` lives in `scroll-command.ts` — extracted from `interaction-commands.ts` for maintainability
- The `index.ts` barrel file in browser-commands/ re-exports all handlers — consumers import from the barrel, so file extractions are transparent
- `withPluginLock(pluginName, fn)` in tab-state.ts serializes async operations per plugin via promise chaining — both `checkTabChanged`/`checkTabRemoved` and `updateLastKnownState` go through this lock
- `checkTabRemoved(tabId)` and `checkTabChanged(tabId, changeInfo)` in tab-state.ts replace the old `checkTabStateChanges` — distinct signatures avoid the boolean `removed` flag
