## Codebase Patterns
- Doctor checks use `pass()`, `warn()`, and `fail()` factory functions that return `CheckResult` objects
- `CheckResult` has `ok` (boolean), `fatal` (boolean), `label`, `detail`, and optional `hint` fields
- `pass()` creates `ok: true, fatal: false`; `warn()` creates `ok: false, fatal: false`; `fail()` creates `ok: false, fatal: true`
- Doctor tests are in `platform/cli/src/commands/doctor.test.ts` — uses `OPENTABS_CONFIG_DIR` env var for test isolation
- The `checkPlugins` function is exported and tested directly (unit tests, not integration)
- The /health endpoint returns `pluginDetails[]` with `source: 'npm' | 'local'` and `failedPlugins[]` with `{ path, error }`
- `handleDoctor` already fetches health data via `checkServerHealth` — reuse `healthData` for any server-dependent checks
- Doctor check functions that read filesystem paths should accept optional params for testability (avoids mocking homedir/cwd)
- Never use `.padEnd()`/`.padStart()` on ANSI-colorized strings — pad raw strings first, then colorize (or colorize first and append spaces manually)

## 2026-02-21 13:05 - US-004
- Fixed ANSI column alignment in `audit.ts` table output
- The issue: `padEnd(statusCol)` was applied to an ANSI-colorized string, so escape codes inflated the character count and produced incorrect visual padding
- Fix: apply color to the icon character first, then append whitespace separately with `' '.repeat(Math.max(0, statusCol - 1))`
- Files changed:
  - `platform/cli/src/commands/audit.ts` (line 125: split into 2 lines)
- **Learnings for future iterations:**
  - Never call `.padEnd()` / `.padStart()` on ANSI-colorized strings — ANSI escape codes are invisible but count as characters. Always pad the raw string first, then colorize, or colorize first and add spaces manually
  - The `statusCol` value (4) accounts for the icon (1 char) plus padding (3 spaces). Since `✓`/`✗` are single visual characters, `statusCol - 1` gives the correct trailing space count
