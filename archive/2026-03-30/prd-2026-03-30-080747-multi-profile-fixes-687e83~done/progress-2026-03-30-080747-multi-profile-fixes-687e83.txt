## Codebase Patterns
- resolveConnection in extension-protocol.ts is a private const — tests must go through dispatchToExtension
- Multi-connection tests use createMockConnection helper (defined at ~line 3408 in extension-protocol.test.ts) to create mock connections with trackable WsHandle
- Test pattern for dispatch routing: call dispatchToExtension, check which mock WS has messages in .sent array, then settle pending dispatch to prevent timeout leaks
- DispatchOptions can be a string (label) or an object — dispatchToExtension normalizes both
- TabMapping.state is 'ready' | 'unavailable' | 'closed' (from state.ts TabState type)
- Browser tool description/summary changes require updating the catalog at platform/shared/src/generated/browser-tools-catalog.ts — the build validates catalog matches definitions at import time
- dispatchToAllConnections returns `{ connectionId: string; result: unknown }[]` — use it for tools that should aggregate across all profiles
- Adding a required field to ExtensionConnection requires updating 12+ test files that construct mock connections — every createMockConnection helper and inline object literal needs the new field
- Raw WS connections can act as fake extensions in E2E tests: they receive JSON-RPC requests (e.g., tool.dispatch) and respond with results using `{ result: { output: <tool result> } }` wrapper
- waitForWsMessage predicates must filter by specific msg.method to avoid matching sync.full notifications sent on connect
- plugin_list_tabs reads directly from server-side tabMapping (no extension round-trip), so no dispatch handler is needed
- setupAutoResponder pattern: for dispatchToAllConnections tests, set up persistent message handlers that auto-respond to specific JSON-RPC methods — cleaner than waitForWsMessage when both WS connections receive independent requests simultaneously
