## Codebase Patterns
- `.catch(() => {})` inside `Promise.allSettled()` maps all rejections to fulfilled results, making any subsequent `result.status === 'rejected'` check dead code. Remove the catch and let allSettled capture rejections directly.
- `platformExec` from `@opentabs-dev/shared` wraps command names for Windows compatibility (e.g., `npm` ‚Üí `npm.cmd`). Always use it instead of bare command strings.
- The `spawnAsync` helper in `plugin-management.ts` (line 64) demonstrates the canonical async spawn pattern with kill handle used in the MCP server.
- When mocking `child_process.spawn` in tests, use `PassThrough` streams (not `Readable.from`) for stdout/stderr to ensure data listeners are attached before data is pushed. Push data and emit 'close' via `process.nextTick`.
- `checkForUpdates` in `version-check.ts` is called from both `reload.ts` (performReload) and `plugin-management.ts` (checkPluginUpdates). Both callers are async and must `await` the result.
- Mutex pattern for serialized async writes: `const p = (async () => { await prev; ... })(); state.mutex = p.catch(() => {}); await p;` ‚Äî the catch ensures the mutex chain always fulfills (no deadlock), while `await p` propagates errors to the caller.
