## Codebase Patterns
- Shared types (ManifestTool, WireToolDef, etc.) live in platform/shared/src/index.ts — this is the single source of truth for the wire protocol
- Optional fields on interfaces use `field?: type` with a JSDoc comment
- The `group` field follows the same optional pattern as `icon`, `iconSvg`, `iconInactiveSvg`
- Wire protocol optional field pattern: parse with `typeof obj.field === 'string' ? obj.field : undefined`, then spread-conditionally include: `...(field ? { field } : {})`
- `serializePluginForExtension` in extension-handlers.ts has an inline return type annotation — add new optional fields to both the type annotation and the implementation
- message-router.ts parses WirePluginDef from raw JSON using defensive type checks — new optional string fields use `typeof t.field === 'string'` guards
- Plugins are standalone npm packages — they install from npm registry, not workspace. In worktrees, override local SDK/plugin-tools by copying dist/ into plugin node_modules
- Tool `group` field goes after `icon` in `defineTool()` calls; uses capitalized plural domain entity names (e.g., "Messages", "Pull Requests")
