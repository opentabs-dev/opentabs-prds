## Codebase Patterns
- The MCP server resolves its package root via `resolve(dirname(fileURLToPath(import.meta.url)), '..')` (same pattern as version.ts)
- Extension→server JSON-RPC handlers follow the pattern: validate params, do work, sendToExtension with result or sendJsonRpcError
- Extension→server methods are dispatched in extension-protocol.ts's handleExtensionMessage function
- New handlers are added to extension-handlers.ts and exported, then imported and routed in extension-protocol.ts
- sync.full in sendSyncFull manually picks fields from configState — new ConfigStateResult fields must be added there too
- The 'open' npm package (sindresorhus/open, v11) is a pure ESM package — use `import open from 'open'` directly
- New bg:* message types need 6 touchpoints: (1) interface in extension-messages.ts, (2) InternalMessage union, (3) bridge function in bridge.ts, (4) handler in background-message-handlers.ts, (5) backgroundHandlers map, (6) EXTENSION_ONLY_TYPES set
- New ConfigStateResult fields need 6 extension layers to flow to the side panel: ServerStateCache, message-router (sync.full + plugins.changed), handleBgGetFullState response, bridge.ts FullStateResult, App.tsx state
