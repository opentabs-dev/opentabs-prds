## Codebase Patterns
- ConfigStatePlugin (shared/src/index.ts) is the canonical type for plugin state flowing to the side panel. Fields added here auto-flow through bridge.ts PluginState via Omit<ConfigStatePlugin, 'tools'>.
- Tab state is cached as serialized JSON (PluginTabStateInfo: { state, tabs }) in the lastKnownState map. Parse it in handleBgGetFullState to extract both aggregate state and per-tab details.
- The PluginTabInfo type (shared/src/index.ts) already existed for tab.stateChanged/tab.syncAll messages.
- Background message handlers follow the `MessageHandler` type pattern: `(message, sendResponse) => void`, with async work wrapped in `(async () => { ... })().then(result => sendResponse(result)).catch(...)`.
- New message types require: (1) interface in extension-messages.ts, (2) add to InternalMessage union, (3) handler in background-message-handlers.ts, (4) register in dispatch map + EXTENSION_ONLY_TYPES, (5) bridge function in bridge.ts.
- `findAllMatchingTabs(meta)` returns tabs sorted by rank (active+focused first). Import from `./tab-matching.js`.
- `getPluginMeta(name)` reads PluginMeta from in-memory cache backed by chrome.storage.local. Import from `./plugin-storage.js`.
- E2E tests that look for PluginIcon border classes (`[class*="border-border/30"]`) must use `xpath=..` to traverse from the accordion trigger to the parent header, because the icon was moved outside the trigger in US-003.
- Plugin `homepage` is read from `pkg.opentabs.homepage` in the MCP server loader and flows via PluginMeta → chrome.storage.local → side panel. The `override` modifier is required when overriding `homepage` in the plugin class.
- `context.waitForEvent('page')` detects tabs created by `chrome.tabs.create` in the extension background. `context.pages()` may not include them.
