## Codebase Patterns
- confirmation-badge.ts uses module-level Maps for state tracking with a count variable kept in sync manually (not derived from map size, because clearConfirmationBadge can be called without an id)
- The test file uses `vi.hoisted` + `vi.mock` pattern for mocking Chrome APIs and module dependencies, with `await import()` for the module under test
- `clearAllConfirmationBadges()` is used in `beforeEach` to reset internal module state between tests
- PendingConfirmationParams interface in confirmation-badge.ts matches ConfirmationData type in ConfirmationDialog.tsx (same shape)
- When adding new exports to a module, all vi.mock() calls for that module in test files must be updated to include the new export — otherwise the import resolves to undefined and causes the async handler to throw into the catch branch
- useServerNotifications.handleNotification accepts an optional receivedAt in params for hydrated confirmations; the auto-removal timeout is computed from elapsed time since receivedAt
- Dedup in handleNotification uses timeoutIds ref (not state) to skip already-registered confirmations — this is more reliable than checking React state which may be stale
