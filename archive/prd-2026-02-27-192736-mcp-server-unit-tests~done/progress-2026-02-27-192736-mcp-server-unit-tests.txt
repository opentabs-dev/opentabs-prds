## Codebase Patterns
- Test files in platform/mcp-server/src/ follow: import the module under test first, then `vitest`, then other deps (import-x/order rule)
- The tsconfig.test.json in mcp-server includes `src/**/*.test.ts` — new test files auto-covered, no tsconfig changes needed
- Functions in mcp-tool-dispatch.ts are declared as `const` (arrow functions), exported at the bottom of the file in a grouped `export { ... }` block
- Run `npm run lint:fix` after creating new test files to auto-fix import order and prettier formatting issues
- Zod v4 is in use (`^4.3.6`); ZodError.issues array with path/message is still available; don't hardcode exact Zod error messages in tests (they may differ from v3)
- Depth limit in sanitizeOutput: `depth > 50` (returns '[Object too deep]' at depth=51+); the function accepts optional depth param from tests
- vi.mock hoisting works safely in the same file as non-mocked tests: pure function tests that don't call mocked modules are unaffected
- Use `vi.mocked(fn)` to access mock state rather than `vi.importMock` (which doesn't exist in Vitest)
- ESLint forbids `!` non-null assertions — use type casts like `(arr[0] as { text: string }).text` or `(arr[0] as unknown[])[1] as Record<string, unknown>` instead
- `handleBrowserToolCall` and `handlePluginToolCall` are already exported from mcp-tool-dispatch.ts; no changes needed
- For mocking the handler's dependencies: mock `./extension-protocol.js`, `./permissions.js`, `./state.js`, `./sanitize-error.js`, `./logger.js` — then import them normally (vi.mock is hoisted)
