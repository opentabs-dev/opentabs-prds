## Codebase Patterns
- Platform packages are in platform/, plugins in plugins/
- extension-protocol.ts is the main WebSocket protocol module; extracted file I/O into adapter-files.ts
- Functions that only depend on config/logger/shared (no ServerState) are good extraction candidates
- Re-export extracted functions from the original module to maintain existing import paths for callers
- knip checks for unused exports, so only re-export what callers actually import
- sendToExtension is defined in extension-handlers.ts and imported by extension-protocol.ts (shared utility)
- McpCallbacks type is exported from extension-handlers.ts and re-exported from extension-protocol.ts for existing callers
- handleExtensionMessage stays in extension-protocol.ts as a thin router delegating to handler functions
- mcp-tool-dispatch.ts contains handleBrowserToolCall and handlePluginToolCall; mcp-setup.ts re-exports sanitizeOutput from it
- import-x/exports-last rule requires all export statements at the end of the file — define as non-exported const, then export at bottom
- Re-exporting types requires `export type { Foo }` at the bottom, not inline `export type` re-export statements at the top
- New fields on ServerState are auto-patched on hot reload (index.ts fills missing keys from createState() defaults) — no STATE_SCHEMA_VERSION bump needed for additions
- browser-tool-names.ts contains the static BROWSER_TOOL_NAMES array; browser-tools/index.ts validates it matches actual definitions at module load time
- import-x/order rule: parent imports (`../`) must come before sibling imports (`./`), but after external package imports
- When moving a module-level flag to ServerState, propagate the state parameter through all intermediate functions that use the flag (e.g., ensureAdaptersDir → writeExecFile → callers)
