## Codebase Patterns
- Browser tools follow a consistent pattern: `defineBrowserTool()` factory with Zod schema and handler
- The analyze-site modules are pure analysis functions (no browser tool dispatch) — the orchestrator (US-005) collects data and passes it in
- Network capture redacts sensitive headers (authorization, cookie, x-csrf-token, x-xsrf-token) to `[REDACTED]` — detection must work with redacted values
- ESLint forbids non-null assertions (`!`) — use explicit type guards (`if (!x) throw` or `if (!x) return`) instead
- Test helpers like `findMethod()` that throw on missing values are the pattern for narrowing `find()` results in tests
- New files in `platform/mcp-server/src/` are automatically covered by the existing tsconfig for both build and test
- ESLint `arrow-body-style` requires removing block braces from single-expression arrow functions
- ESLint `@typescript-eslint/no-unnecessary-condition` flags `typeof x === 'object'` checks on values already typed as Record — remove them
- The `NetworkRequest` type from detect-auth.ts is the shared type used across all analyze-site modules
- For import ordering in test files, type imports must be alphabetically ordered by module path (`./detect-apis.js` before `./detect-auth.js`)
- Framework and globals detection modules follow the same pure-function pattern as auth/APIs: take structured input, return structured output
- `FrameworkDetectionInput` uses pre-computed booleans (hasSingleRootElement, usesPushState, hasNextData, etc.) — the page-context probing happens in the orchestrator, not the detection module
- `GlobalsDetectionInput` uses pre-collected `GlobalProperty[]` with path, type, and topLevelKeys — the heavy Object.keys(window) filtering happens in a page script, this module just classifies
- Auth-related key pattern for globals: `/session|token|auth|jwt|csrf|user|login|credential|account|profile/i`
- The orchestrator dispatches to the extension using `dispatchToExtension()` directly — not through browser tool handlers — to avoid redundant Zod validation
- Import ordering: value imports first (alphabetical by module path), then type imports (alphabetical by module path); merge all named imports from the same module into one statement
- `browser.getCookies` takes a `url` param (not `tabId`); returns `{ cookies: [...] }` not a raw array
- `browser.getNetworkRequests` result may be `{ requests: [...] }` or a raw array — handle both shapes
- `browser.openTab` returns `{ id, title, url, windowId }` — use `.id` not `.tabId`
- `browser.executeScript` results are nested `{ value: { value: <actual>, error?: <msg> } }` — must unwrap
- E2E test filters for "plugin-only tools" should use BROWSER_TOOL_NAMES set, not prefix matching
- Analyze-site test servers use relative URLs in client-side JS to avoid TypeScript circular reference errors with `server.port`
- Network capture can include noise requests (Chrome extension internals, check calls) — E2E tests should assert specific expected suggestions rather than strict validation on all suggestions
- Page-context scripts for `browser.executeScript` must be plain JS strings (the MAIN world does not run TypeScript)
- SESSION_COOKIE_PATTERNS use exact-match regexes — test server cookies must use names that exactly match (e.g., `session`, `connect.sid`, `sid`), NOT names like `session_id` which fall between patterns
- Chrome CDP fires `Network.webSocketCreated` (not `Network.requestWillBeSent`) for WebSocket connections — the network-capture module handles this as a synthetic request with status 101
- Network capture preserves auth scheme prefix in Authorization headers: `"Bearer [REDACTED]"` not `"[REDACTED]"` — enables detect-auth to distinguish Basic from Bearer
