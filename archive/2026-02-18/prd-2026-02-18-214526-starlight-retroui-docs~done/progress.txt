## Codebase Patterns
- docs/ is a standalone project (like plugins/) — NOT in root bun workspaces or tsconfig references
- docs/ must be added to eslint ignores and knip ignores (same as plugins/)
- Root .gitignore already covers `node_modules/` and `dist/` recursively (no need for explicit docs/ entries)
- docs/ uses `astro/tsconfigs/strict` for its tsconfig
- Starlight Tailwind v4 setup: `@layer base, starlight, theme, components, utilities;` then `@import '@astrojs/starlight-tailwind';` + `@import 'tailwindcss/theme.css' layer(theme);` + `@import 'tailwindcss/utilities.css' layer(utilities);`
- CSS `@import url(...)` for Google Fonts causes ordering warnings in Tailwind v4 pipeline — use Starlight `head` config with `<link>` tags instead
- Starlight uses `[data-theme='dark']` for dark mode, RetroUI uses `.dark` — apply dark tokens to both selectors
- `@astrojs/starlight-tailwind` provides `@custom-variant dark` for `[data-theme=dark]` and maps Tailwind colors to `--sl-*` variables via a `@layer utilities` block
- astro.config.mjs needs `vite: { plugins: [tailwindcss()] }` with `@tailwindcss/vite` for Tailwind v4
- Astro 5 requires `src/content.config.ts` with `docsLoader()` and `docsSchema()` from Starlight — auto-generation is deprecated and can cause stale data store issues
- Stale `node_modules/.astro/data-store.json` can cause content collection entries to not appear in production builds — delete it and rebuild if pages are missing
- VitePress links use `/guide/name` format; Starlight uses `/guide/name/` (trailing slash) for internal links
- Starlight generates page titles from frontmatter `title:` field — remove `# Title` H1 headings from content to avoid duplication
