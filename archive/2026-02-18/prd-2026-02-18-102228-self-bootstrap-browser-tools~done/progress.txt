## Codebase Patterns
- Network capture state lives in `platform/browser-extension/src/network-capture.ts` — uses a `CapturedRequest` interface and per-tab `CaptureState` stored in a `captures` Map
- CDP events flow through a single `chrome.debugger.onEvent` listener; new fields on `CapturedRequest` are extracted from CDP event params and carried from `pendingRequests` to the completed `requests` array
- Response bodies arrive via `Network.loadingFinished` (fires AFTER `Network.responseReceived`), requiring a `requestIdToIndex` map to locate the completed request and attach the body asynchronously via `chrome.debugger.sendCommand` callback
- Binary MIME types (image/*, font/*, video/*, audio/*, application/octet-stream, application/wasm) are skipped for response body capture
- When `requests.shift()` drops the oldest entry, all indices in `requestIdToIndex` must be decremented by 1 (entries at index 0 are removed)
- Tool descriptions live in `defineBrowserTool` calls in `platform/mcp-server/src/browser-tools/*.ts`
- Body truncation uses a constant `MAX_BODY_LENGTH = 102_400` (100KB) with `'... (truncated)'` suffix
- The `requestBody` data comes from `request.postData` in CDP `Network.requestWillBeSent` event — no additional CDP commands needed
- Adding a new browser tool requires 5 touchpoints: (1) `get-*.ts` tool definition, (2) `index.ts` import+array, (3) `browser-commands.ts` handler, (4) `message-router.ts` dispatch table+import, (5) `offscreen/index.ts` ALLOWED_METHODS
- The `message-router.test.ts` uses `mock.module` to replace entire modules — when adding new exports to `browser-commands.ts`, the mock must also include the new export or the test will fail with "Export not found"
- Browser tool handlers in `browser-commands.ts` follow a consistent pattern: validate params, `chrome.scripting.executeScript` with `world: 'MAIN'`, extract result, send JSON-RPC response via `sendToServer`

- **Learnings for future iterations:**
  - The `chrome.scripting.executeScript` `func` runs in the page's MAIN world — it can access `window.localStorage` and `window.sessionStorage` directly
  - For tools that have two return modes (single key vs all entries), use a `mode` discriminator field in the script result to let the handler know which shape to return
  - `Object.keys(storage)` works for iterating localStorage/sessionStorage keys — more portable than `storage.key(i)`
  - Value truncation (10KB per value) and total response cap (500KB) are separate concerns — both are needed to handle pages with many small entries vs pages with a few large entries
