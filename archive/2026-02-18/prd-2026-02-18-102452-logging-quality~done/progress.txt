## Codebase Patterns
- Browser extension log prefixes follow `[opentabs]` or `[opentabs:<context>]` pattern (e.g., `[opentabs:background]`, `[opentabs:offscreen]`)
- Console calls injected into target tabs via `chrome.scripting.executeScript` (IIFE teardown, tool invocation logs) must also use the `[opentabs]` prefix
- Use `console.warn` (not `console.error`) for expected lifecycle failures in the browser extension
- Catch callback parameters must use `: unknown` type annotation (ESLint: `@typescript-eslint/use-unknown-in-catch-callback-variable`)
- MCP server logger (`platform/mcp-server/src/logger.ts`) is the single formatting point for all server log output — log format is `[opentabs] <ISO timestamp> <message>`
- Pass raw Error objects to log.error/log.warn — the logger's formatArgs helper extracts message (all levels) and stack (debug only). Never use `err instanceof Error ? err.message : String(err)` in log calls.
- The `err instanceof Error ? err.message : String(err)` pattern is still valid for constructing error messages for HTTP responses, MCP tool responses, and Error constructors — just not for log calls.
- Build scripts use `[opentabs:build:<target>]` prefix (e.g., `[opentabs:build:background]`, `[opentabs:build:side-panel]`) — all platform output is filterable with `[opentabs`
