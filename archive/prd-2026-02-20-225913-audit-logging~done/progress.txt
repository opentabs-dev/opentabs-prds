## Codebase Patterns
- ServerState fields survive hot reloads via globalThis — new fields on state are immediately available to all modules
- STATE_SCHEMA_VERSION must be bumped when adding new fields to ServerState
- Browser tools use a separate code path (cachedBrowserTools) from plugin tools in the tools/call handler
- Plugin tool error info comes from DispatchError (with isDispatchError check) which has code, message, and data fields; generic errors use code 'UNKNOWN'
- The tools/call handler uses a finally block for cleanup (activeDispatches, sendInvocationEnd); audit logging fits naturally there
- HTTP routes in http-routes.ts use a sequential if/return pattern with checkBearerAuth for protected endpoints
- state.auditLog stores entries in chronological order (oldest first); reverse for newest-first API responses
- CLI commands follow registerXxxCommand(program) pattern in platform/cli/src/commands/ with export from index.ts and registration in cli.ts
- Global --port option is accessed via command.optsWithGlobals() — subcommand options interfaces must include port?: number
- Secret for Bearer auth is read from ~/.opentabs/config.json via readConfig() + config.secret
- Connection refused detection uses isConnectionRefused() helper from config.ts
- E2E tests use standard Playwright fixtures (mcpServer, testServer, extensionContext, mcpClient) from fixtures.ts
- E2E test HTTP requests to the MCP server use mcpServer.port and mcpServer.secret for auth
- E2E tests must avoid non-null assertions (use guard clauses with throw instead)
