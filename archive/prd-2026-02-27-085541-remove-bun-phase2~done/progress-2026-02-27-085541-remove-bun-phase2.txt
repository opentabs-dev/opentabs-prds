## Codebase Patterns
- Dev proxy (platform/mcp-server/src/dev-proxy.ts) logs "Dev proxy listening" BEFORE starting the worker. Wait for "Worker ready" before treating the server as ready.
- Dev proxy restarts the worker process on SIGUSR1. MCP sessions don't persist across worker restarts — the MCP client auto-reinitializes on 400/502/503.
- In-flight HTTP requests through the proxy may fail with 502 when the worker is killed during hot reload. This is expected proxy behavior.
- The auth secret (auth.json) is written by the worker during startup. Don't read it until the worker is confirmed ready.
- E2E test fixtures.ts uses `hot` param in `startMcpServer` — hot=true spawns dev-proxy.js, hot=false spawns index.js directly.
- npm run commands work from root; bun run fails for npm --prefix subcommands.

- **Learnings for future iterations:**
  - The root cause of 56 failing E2E tests was that `startMcpServer` resolved when the dev proxy logged "listening" but the worker hadn't started yet. The auth secret (auth.json) wasn't written yet, causing all authenticated requests to fail with 401.
  - The proxy model differs from bun --hot: bun --hot keeps the process alive and reloads modules (sessions survive), while the proxy kills and restarts the worker (sessions are lost, in-flight requests may 502).
  - Tests that assert on server log messages about session re-registration don't apply with the proxy model.
  - The flaky "old extension WS" test was caused by a single health() call after waitForExtensionConnected — a brief disconnect-reconnect cycle could cause the single call to see stale state. Using waitForHealth polling instead is more robust.
