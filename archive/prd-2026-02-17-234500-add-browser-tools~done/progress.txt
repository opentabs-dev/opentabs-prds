## Codebase Patterns
- Browser tool MCP server files go in `platform/mcp-server/src/browser-tools/` using `defineBrowserTool` from `./definition.js`
- Each tool dispatches to extension via `dispatchToExtension(state, 'browser.methodName', params)`
- Extension handlers in `platform/browser-extension/src/browser-commands.ts` follow pattern: validate params, call chrome API, sendToServer with result/error
- Message router dispatch table in `platform/browser-extension/src/message-router.ts` uses `methodHandlers` Map
- Sync handlers (no `await`) must NOT be `async` — the dispatch table calls `.catch()` on async handlers but directly invokes sync ones
- `chrome.debugger.onEvent` callback param type is `object | undefined` — cast to `Record<string, unknown>` inside the handler
- `chrome.debugger.detach()` returns a Promise — use `void ...catch(() => {})` for fire-and-forget detach
- BROWSER_TOOL_NAMES in e2e/helpers.ts must only list tools that exist on the MCP server, not future tools
- All handlers wrapped with `if (id !== undefined) { handler(params, id).catch(console.error); }`
- URL validation uses `safeUrl` from `./url-validation.js` (allowlist http/https only)
- Extension-side URL validation uses `isBlockedUrlScheme` from `@opentabs-dev/shared`
- Browser tools barrel: `platform/mcp-server/src/browser-tools/index.ts` — add new tools to the `browserTools` array
- E2E test helpers in `e2e/helpers.ts`, browser tool names in `BROWSER_TOOL_NAMES` array
- Test server in `e2e/test-server.ts` serves HTML pages and API endpoints for plugin testing
- `chrome.tabs.update()` returns `Tab | undefined` — always guard against undefined before accessing properties
- Bun `mock.module` has a bug: `mock<Type>()` (typed mock without implementation) causes named exports to resolve as `undefined` in consuming modules' static import bindings. Workaround: use `mock(asyncNoop as Type)` with an actual implementation function
- When adding new browser command handlers, also update the mock.module factory in `message-router.test.ts`
- For DOM interaction tools, use `chrome.scripting.executeScript({ target: { tabId }, world: 'MAIN', func, args })` — returns `results[0].result`
- Return `{ error: string }` from the func for element-not-found cases; check before sending JSON-RPC response
- Cookie tools use `safeUrl` from `url-validation.ts` for server-side URL validation + `isBlockedUrlScheme` for extension-side defense-in-depth
- `chrome.cookies.set()` returns `Cookie | null` — always guard against null
- **CRITICAL**: When adding new browser tool methods, also add them to `ALLOWED_METHODS` in `platform/browser-extension/src/offscreen/index.ts` — the offscreen document drops WebSocket messages with unrecognized methods, causing tool dispatch to silently hang until the 30s timeout
