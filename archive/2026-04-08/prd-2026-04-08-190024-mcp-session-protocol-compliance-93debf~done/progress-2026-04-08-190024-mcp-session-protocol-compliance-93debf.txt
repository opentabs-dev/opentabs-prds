## Codebase Patterns
- `handleMcp` and `handleGatewayMcp` in `platform/mcp-server/src/http-routes.ts` have nearly identical code structures — changes to one typically need mirroring in the other
- POST flow: existing session lookup → rate limit → parse body → initialize check → stale session 404 / missing session 400
- GET/DELETE flow: missing sessionId → 405, valid session → handleRequest, stale session → 404
- The rate limit check runs before the stale session check for POST, so 429 takes priority over 404
- Existing tests in `http-routes.test.ts` cover rate limiting with unknown session IDs (expects 429 since rate limit fires first)
- For E2E protocol compliance tests, use `startMcpServer(configDir, false)` to bypass the dev proxy and test handlers directly
- `createMcpClient` targets `/mcp` — for `/mcp/gateway` tests, use raw `fetch()` calls
- Fabricate `randomUUID()` as stale session IDs rather than restarting servers
