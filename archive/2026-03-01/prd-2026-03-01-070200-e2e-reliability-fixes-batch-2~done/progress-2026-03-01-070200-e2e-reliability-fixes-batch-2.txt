## Codebase Patterns
- E2E tests are in `e2e/` at the repo root (not in individual packages)
- Shared test fixtures are in `e2e/fixtures.ts`; helpers like `waitForLog`, `callToolExpectSuccess`, `setupToolTest` are in `e2e/helpers.ts`
- The `mcpServer` fixture exposes `.health()` and `.waitForHealth()` for polling plugin buffer state
- Plugin log buffer sizes are accessible via `health.pluginDetails?.find(p => p.name === '...')?.logBufferSize`
- For shared buffers under parallel test concurrency, use a range assertion `[n, n+k]` rather than exact equality
- `@typescript-eslint/no-non-null-assertion` is ENFORCED — never use `!.` to access properties; instead redesign to use proper guards (e.g., take a path param instead of a URL so construction happens after the guard)
- Never use `server.logs.length = 0` before `waitForLog` — this races with log lines arriving between truncation and the wait call. Instead, count occurrences before the action and use `waitFor(() => count > before)` to detect a NEW occurrence
- When using occurrence-counting for multiple log lines triggered by the same action, capture ALL baseline counts BEFORE triggering the action — log lines may arrive in any order, so capturing a second baseline after waiting for the first log can miss occurrences emitted before the first log

- **Learnings for future iterations:**
  - The throw-inside-try antipattern (throwing inside try with a catch that asserts on the error message) is dangerous: if the throw message doesn't match the catch's regex, you get a confusing failure about the regex not matching rather than a clear "operation succeeded when it shouldn't have"

  - ESLint's `no-throw-literal` and similar rules don't catch this specific antipattern, so it requires manual review
