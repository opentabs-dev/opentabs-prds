## Codebase Patterns
- export-har.ts uses a two-phase fetch design: fetch-without-clear first (to protect against partial data loss if a subsequent fetch fails), then clear separately
- The clearing dispatch to `browser.getNetworkRequests` with `clear:true` returns the COMPLETE buffer (all entries including any that arrived after the non-clearing fetch), making the replace-instead-of-merge approach safe
- The `entries` array is typed as `const` but is a mutable array; `entries.length = 0` clears it in place without reassignment

## 2026-03-01 - US-001
- Implemented fix for silent data loss in export-har buffer clearing
- Modified `platform/mcp-server/src/browser-tools/export-har.ts` lines 223-250
- **Problem**: When `args.clear` was true, the clearing dispatches' responses were discarded. Requests arriving between the initial fetch and the clear were silently lost.
- **Fix**: Capture the responses from clearing dispatches. Since the clearing fetch returns the complete buffer (old + new entries), replace `entries` array contents with the clearing fetch's results instead of merging. This avoids deduplication complexity entirely.
- **Approach**: Used `entries.length = 0` to clear the array, then `entries.push(...)` to repopulate with clearing fetch results. For WebSocket frames, same pattern applied inside the `args.includeWebSocketFrames` block.
- All Phase 1 checks passed (build, type-check, lint, knip, test — 2890 tests)
- **Learnings for future iterations:**
  - The replace-instead-of-merge approach works because `getNetworkRequests` without clearing returns a snapshot of the current buffer, and the subsequent clear call returns the same snapshot plus any new entries — no overlap possible since the buffer is an append-only log
  - `entries` is a `const` array but mutation is fine — TypeScript's `const` only prevents reassignment
