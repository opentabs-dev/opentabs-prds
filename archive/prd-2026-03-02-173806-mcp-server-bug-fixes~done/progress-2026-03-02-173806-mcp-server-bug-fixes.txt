## Codebase Patterns
- sanitize-error.ts chains .replace() calls in a specific order: URLs first, then paths, then localhost, then IPs. New IP-family regexes must go BEFORE IPv4 to avoid partial matches.
- IPv6 compressed-form regex requires separating colon-terminated middle groups from the final pre-:: group: `(?:[0-9a-fA-F]{1,4}:)*[0-9a-fA-F]{0,4}::` — greedy `hex:*` alone eats the colon before `::` and breaks the match.
- Bracket-wrapped IPv6 regex `\[[0-9a-fA-F]*:[0-9a-fA-F:]*\]` requires at least one colon inside brackets to avoid false-positives on array indices like `[0]`.
- Shared browser-tools utilities go in `platform/mcp-server/src/browser-tools/` (e.g., dispatch-utils.ts). Files in analyze-site/ import with `../`, files in browser-tools/ import with `./`.
- The ESLint import-x/order rule requires `./` (same-dir) imports to appear before `../` (parent-dir) imports — add new imports in the right order or run `npm run lint:fix`.

## 2026-03-03 - US-001
- Added IPv6 address sanitization to platform/mcp-server/src/sanitize-error.ts
- Added bracket-wrapped regex `/\[[0-9a-fA-F]*:[0-9a-fA-F:]*(?:%[^\]]+)?\]/g` before IPv4 regex
- Added compressed-form regex `/(?:[0-9a-fA-F]{1,4}:)*[0-9a-fA-F]{0,4}::(?:[0-9a-fA-F:.]+)?/g` before IPv4 regex
- Added 9 IPv6 unit tests in sanitize-error.test.ts covering: [::1], [fe80::1], [2001:db8::1], ::1, fe80::1, 2001:db8::1, ::ffff:192.168.1.1, plus false-positive guards for [0] and version strings
- **Learnings for future iterations:**
  - The greedy `(?:[hex]{1,4}:)*::` pattern fails for `fe80::1` because the `*` group eats `fe80:` (consuming the first colon of `::`) leaving only `:1` — not `::`. Fix: use `(?:[hex]{1,4}:)*[hex]{0,4}::` to separate middle groups from the immediate pre-:: group.
  - Mixed IPv6/IPv4 like `::ffff:192.168.1.1` is fully consumed by the IPv6 regex because `(?:[0-9a-fA-F:.]+)?` includes dots, so no IPv4 residue leaks.
