## Codebase Patterns
- `platform/plugin-sdk/src/dom.ts` exports DOM utilities (waitForSelector, waitForSelectorRemoval, querySelectorAll, getTextContent, observeDOM) used by plugin authors
- MutationObserver callbacks fire asynchronously — TypeScript control flow analysis knows boolean flags (like `settled`) set before `observer.observe()` are still at their original value in code immediately following it
- `@typescript-eslint/no-unnecessary-condition` will flag `if (!settled)` after `observer.observe()` because settled is statically always false there; remove the outer guard and rely on the settled flag in the observer callback to prevent double-resolves

- **Learnings for future iterations:**
  - The `@typescript-eslint/no-unnecessary-condition` lint rule will flag `if (!settled)` immediately after `observer.observe()` because TypeScript knows `settled` is always `false` at that point (MutationObserver callbacks are async). The correct fix is to omit the outer `!settled` guard — the observer callback itself already guards with `if (settled) return;` to prevent double-resolves.
  - The re-check pattern (try/catch around querySelector, swallowing errors) is correct because the observer callback already handles errors from subsequent querySelector calls. The re-check only needs to handle the success path.
  - Test mocking approach: `vi.spyOn(document, 'querySelector')` with a mock that returns null on call 1 (initial check) and an element on call 2 (re-check) proves the TOCTOU fix without needing any DOM mutations to fire.
