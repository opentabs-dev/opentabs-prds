## Codebase Patterns
- Side panel pure-function test pattern: extract core logic from hooks into exported pure functions (e.g., `processNotification` from `useServerNotifications`), test those functions directly with mock setters (vi.fn()) and vi.useFakeTimers() for timeout behavior
- Test files are `.test.ts` (not `.test.tsx`) — no jsdom or React rendering needed for pure logic tests
- All imports in test files use `.js` extensions
- Mock `chrome.runtime.sendMessage` in beforeEach as a vi.fn() on globalThis.chrome
- Use `as ReturnType<typeof vi.fn>` to access `.mock.calls` on mock setters
- The `@typescript-eslint/no-non-null-assertion` rule is enforced — use `if (!x) throw` guards instead of `x!`
- State updater mock pattern: create `resolveSetX(prev)` helpers that extract the updater function from the last mock call and invoke it with the given previous state
- Array element access (`arr[0]`) requires a guard (`if (!first) throw`) — TS strict mode flags `arr[0].field` as possibly undefined
- normalizeConfigState pattern: when normalizing unknown data, cast to `Record<string, unknown>` after null/type check, then use Array.isArray guards for each array field; spread plugin objects and override `tools` field
- Use `matchesPlugin` (from bridge.ts) for plugin list filtering — it checks plugin name, displayName, AND tool names; the simpler inline `p.tools.some(t => matchesTool(...))` only checks tools
