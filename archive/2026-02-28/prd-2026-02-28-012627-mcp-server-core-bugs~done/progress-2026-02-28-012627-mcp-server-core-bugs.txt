## Codebase Patterns
- platform/mcp-server/src/reload.ts: reloadCore handles config/registry discovery in one try-catch and loadSecret independently in a separate try-catch
- arrow-body-style ESLint rule: when a function body contains only a return statement, ESLint requires concise arrow syntax (no braces). Triggered when removing statements that made the block body necessary.
- MCP server state fields (registry, toolConfig, browserToolPolicy, etc.) are updated from config/discovery; wsSecret is updated separately from loadSecret
- plugin-management.ts configWriteMutex pattern: async closure chained to prev mutex; catch handler resets mutex to Promise.resolve() and logs; do NOT re-throw or subsequent awaits see rejection instead of false return
- dispatchToExtension accepts optional `timeoutMs` in DispatchOptions to override the default 30s DISPATCH_TIMEOUT_MS; use this for lightweight/best-effort dispatches (e.g., domain resolution)

- **Learnings for future iterations:**
  - The configWriteMutex pattern: `state.configWriteMutex = (async () => {...})().catch(handler)` — the `.catch()` result promise is what line 439 awaits (captured by value at evaluation time). If catch re-throws, the await throws; if catch returns normally, the await resolves.
  - To return false on failure (rather than throwing), the catch handler must NOT re-throw — it should handle the error gracefully (log it, reset mutex) and let the promise resolve
  - The `state.configWriteMutex = Promise.resolve()` reassignment inside the catch is for future chainers (they do `const prev = state.configWriteMutex` before enqueueing themselves), not for the current `await state.configWriteMutex` which already captured the original promise reference
