## Codebase Patterns
- MCP client (createMcpClient in e2e/fixtures.ts) throws `Error: MCP request failed (${status}): ${body}` on non-2xx responses â€” so 401 rejections produce messages containing "401"
- e2e test files use `let varName = ''` initialization for string vars that might not be set in beforeEach (allows guard `if (varName)` in afterEach)
- To verify IIFE re-injection in e2e tests: use `replaceIifeClosing` to inject a `globalThis.__e2e<Name>` marker, then poll with `waitFor` until the marker appears in the page. Adapter properties can't be used because the adapter object is frozen.
- Non-fixture tests that create resources manually should use the `let resource: Type | undefined; try/catch` pattern from `setupIsolatedIifeTest` (helpers.ts) to handle partial-failure cleanup
- `setupAdapterSymlink` (helpers.ts) encapsulates the adapter symlink boilerplate (mkdir + rmSync + symlinkCrossPlatform)
- Tool enable/disable is via WebSocket extension protocol methods `config.setToolEnabled` and `config.setAllToolsEnabled` (NOT MCP tools). The PRD referred to "plugin.enable"/"plugin.disable" but the actual API is `config.setAllToolsEnabled { plugin, enabled }` and `config.setToolEnabled { plugin, tool, enabled }`. These trigger `notifyToolListChanged` so MCP clients see updated tools/list.
- To test tool enable/disable E2E: connect WebSocket via `connectWs` to send config methods, create a separate MCP client via `createMcpClient` to verify `tools/list` changes, and use `waitForToolList` to poll for the change.
