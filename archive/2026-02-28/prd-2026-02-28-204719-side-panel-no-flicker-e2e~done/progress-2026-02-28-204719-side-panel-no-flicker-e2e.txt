## Codebase Patterns
- E2E tests for side panel use `openSidePanel(context)` which navigates to `chrome-extension://<id>/side-panel/side-panel.html` in a regular tab
- The plugin list accordion uses `button[data-radix-collection-item]` as the selector for individual plugin items (not `button[aria-expanded]`)
- Side panel receives plugin updates via two paths: `sp:serverMessage` with `plugins.changed` and `ws:message` with `sync.full`
- `loadPlugins()` calls `fetchConfigState()` which sends `config.getState` to the MCP server via the background script bridge
- The `writeTestConfig` → `waitForLog('Config reload complete')` → `postReload()` pattern ensures the MCP server has processed plugin changes before the side panel checks
- To restart a server on the same port, use `startMcpServer(configDir, true, originalPort)` — the extension's WebSocket reconnect logic picks up the new server automatically
- Side panel App.tsx uses a `pluginsLoaded` state gate to prevent NoPluginsState from rendering before the initial plugin fetch completes
