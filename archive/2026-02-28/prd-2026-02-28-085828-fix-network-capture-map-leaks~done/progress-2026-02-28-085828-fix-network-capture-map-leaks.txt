## Codebase Patterns
- network-capture.ts module-level `captures` Map holds all CaptureState; listeners are registered once at module load
- CaptureState.requestIdToRequest maps requestId → CapturedRequest for body attachment in loadingFinished
- CaptureState.pendingRequests maps requestId → partial request data between requestWillBeSent and responseReceived
- CaptureState.wsFramesByRequestId maps requestId → WebSocket URL; entries are added in webSocketCreated and removed in webSocketClosed or stopCapture
- Tests use unique tabIds per test to avoid state contamination (module-level captures Map)
- Chrome API mocks (debugger, tabs, runtime) are set on globalThis.chrome before the module is dynamically imported
- vi.useFakeTimers() / vi.useRealTimers() work correctly for setInterval tests; vi.getTimerCount() verifies active timer count
- Verify interval was cleared by checking vi.getTimerCount() === 0 after stopCapture
- The pre-commit hook runs prettier and eslint --fix automatically; inline objects that exceed line length get expanded to multi-line by prettier
- All cleanup paths in stopCapture must explicitly clear Maps before captures.delete(tabId) for deterministic memory release

- **Learnings for future iterations:**
  - The interval closes over the local captureState variable rather than re-fetching from the captures map; this ensures the interval always operates on the right state object
  - All three capture-cleanup paths (stopCapture, onRemoved, onDetach) must clearInterval to prevent orphaned timers
  - The pruning condition is `now - timestamp > PENDING_REQUEST_TTL_MS` (strictly greater), so testing requires advancing past TTL: with TTL=60s and interval=30s, a 90s advance is needed (first tick=30s: not stale; second tick=60s: 60>60 is false; third tick=90s: 90>60 is true)
