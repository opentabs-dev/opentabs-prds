## Codebase Patterns
- `handleWsInfo()` in `platform/mcp-server/src/http-routes.ts:239` is the /ws-info endpoint
- CORS check at lines 521-524 runs before the router — only `chrome-extension://` Origins and no-Origin pass through
- DNS rebinding protection at lines 508-511 blocks non-localhost Host headers
- `checkBearerAuth(req, state.wsSecret)` returns `null` on success or a `Response` (401) on failure
- Unit tests in `http-routes.test.ts` use `createTestHandlers()` + `handlers.fetch(req, mockServer)` pattern
- The full request pipeline (including CORS/Host checks) runs when using `handlers.fetch()`
- `/ws-info` response body: `{ wsUrl, wsSecret? }` — wsSecret included when state.wsSecret is set (line 252)
- `bootstrapFromAuthFile()` in offscreen/index.ts:509 — reads auth.json via chrome.runtime.getURL; sets module-level `wsSecret`
- `refreshWsUrl()` in offscreen/index.ts:240 — calls fetchWsInfo, updates wsSecret and mcpServerUrl from response; called before each connect()
- `offscreen/index.test.ts` only tests pure utility functions (`isValidWsOrigin`, `wsToHttpBase`) — the main connection logic is covered by E2E tests
- E2E tests that need custom extension setup (no auth.json, wrong secret) should use `launchExtensionContext(port, undefined)` or `launchExtensionContext(port, 'wrong-secret')` directly instead of the `extensionContext` fixture
- `setupAdapterSymlink()` from `e2e/helpers.ts` handles the adapter dir symlink; auth.json symlink is only in the `extensionContext` fixture
- For no-auth tests: pass `undefined` as secret to skip auth.json creation; do NOT symlink from server configDir
- **CRITICAL: Chrome extension offscreen documents do NOT send `Origin: chrome-extension://` headers with `fetch()` requests.** Service workers also don't send Origin. Only web pages and content scripts send Origin headers. The /ws-info endpoint allows no-Origin requests through (not just chrome-extension:// Origin) because the CORS check already blocks malicious web origins before reaching the endpoint.
