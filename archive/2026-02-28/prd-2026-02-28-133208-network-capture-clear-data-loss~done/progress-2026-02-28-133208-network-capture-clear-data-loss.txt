## Codebase Patterns
- network-capture.ts has two kinds of internal state: *output buffers* (requests, wsFrames, consoleLogs) that are returned to callers, and *lookup maps* (pendingRequests, requestIdToRequest, wsFramesByRequestId, wsCreatedAt) that are needed for ongoing event routing. clear=true should only reset the output buffers, never the lookup maps.
- The body-fetch guard at line ~304 (`if (!state.requests.includes(request)) return`) safely handles the case where a loadingFinished async callback fires after clear — no extra guard needed in getRequests.
- WebSocket entries in wsFramesByRequestId are cleaned up by three mechanisms: webSocketClosed events, the periodic prune interval (WS_TTL_MS = 5 min), and stopCapture. getWsFrames(clear=true) must not interfere with any of these.
- Chrome API is mocked at the top of network-capture.test.ts using globalThis.chrome; the module is loaded via dynamic import AFTER the mock is set up.
- Test tab IDs use ranges to avoid collisions: 1xxx=getRequests, 2xxx=loadingFinished, 3xxx=webSocketClosed/stopCapture, 4xxx=pruning interval, 5xxx=wsFramesByRequestId pruning, 6xxx=concurrent startCapture, 7xxx=getWsFrames clear.
