## Codebase Patterns
- `PluginTabInfo` in `@opentabs-dev/shared` has `tabId`, `url`, `title`, `ready` — `title` is required (not optional); wire format `WirePluginTabInfo` is loose and doesn't require `title`
- `handleTabSyncAll` and `handleTabStateChanged` in `extension-handlers.ts` accept `Record<string, unknown>` params; `WirePluginTabInfo` is used internally for wire parsing (not `PluginTabInfo`)
- For `handleTabStateChanged`, computing `newMapping = parseTabMapping(wire)` before calling `state.tabMapping.set(plugin, newMapping)` avoids a non-null assertion to re-read the just-set value
- `isSessionAllowed` in `platform/mcp-server/src/state.ts` checks session permission rules; `domain_all` scope means "allow all tools on a specific domain" (not "allow all domains")
- Session permission rules are pushed in `handleConfirmationResponse` in `platform/mcp-server/src/extension-handlers.ts`; the `scope` field controls which fields (tool/domain) are null
- Tests for `isSessionAllowed` live in `state.test.ts`; tests for `handleConfirmationResponse` live in `extension-handlers.test.ts`
- Prettier print width causes long strings in function args to be broken onto their own line with trailing comma: `log.warn(\n  \`...\`,\n);`
- `log` is imported from `./logger.js` in extension-handlers.ts
- `rebuildCachedBrowserTools` in `mcp-setup.ts` only reads `state.browserTools` and writes `state.cachedBrowserTools` — a minimal staging object `{ browserTools, cachedBrowserTools }` is sufficient for calling it without touching real state
- `pruneStaleState` in `reload.ts` modifies Maps in-place (`tabMapping`, `activeDispatches`) using `state.registry` to check which plugins still exist; runs after the state swap, not before
- `reload.test.ts` integration tests use a real temp config dir (via `OPENTABS_CONFIG_DIR`) and `(globalThis).__opentabs_global_paths__ = []` to isolate from globally-installed npm plugins
- `file-watcher.ts:parseToolsJson` is a local helper (not exported); `validateResources` and `validatePrompts` are exported from `loader.ts` and accept `(items: unknown[], pluginName: string, sourcePath: string)` for error reporting
- The manifest object format in tools.json is `{ tools: [...], resources: [...], prompts: [...] }`; legacy format is a plain array (tools only); `extractToolsArray` handles both but only returns the tools array
- Bounded array pattern in `state.ts`: use a `MAX_X` constant + a `pushX(state, entry)` helper that trims oldest entries beyond the cap (e.g., `appendAuditEntry`/`MAX_AUDIT_ENTRIES`, `pushSessionPermission`/`MAX_SESSION_PERMISSIONS`)
- `state.registry.toolLookup` provides O(1) lookup by prefixed tool name — use it in `pruneStaleState` for existence checks
- The ws library's `'headers'` event provides `(headers: string[], request: IncomingMessage)` — use the request as a Map key for per-connection state instead of shared variables

  1. `server-node.ts` — Replaced shared `pendingUpgradeHeaders` variable with a `Map<IncomingMessage, Record<string, string>>` (`pendingHeadersByReq`). The `'headers'` event callback uses its second parameter (`request: IncomingMessage`) to look up the correct pending headers for each upgrade.

  3. `server-node.ts` — Separated HTTP adapter from upgrade adapter: HTTP requests use a shared `httpAdapter` (with `upgrade: () => false`), while each WebSocket upgrade gets a fresh `perRequestAdapter`. The previous single shared `adapter` was both unnecessary and a race vector.
  4. Added `UpgradeContext` interface to name the upgrade state type explicitly.
- **Files changed**: `platform/mcp-server/src/server-node.ts`
- **Learnings for future iterations:**
  - The ws library's `'headers'` event signature is `(headers: string[], request: IncomingMessage)` — the second parameter provides the HTTP request, enabling per-request header lookup via a `Map<IncomingMessage, ...>` keyed by the request object.
  - TypeScript's control flow analysis narrows `let x: T | null = nonNull` to `T` if no `null` assignment is reachable before the read — using `x?.prop` triggers `@typescript-eslint/no-unnecessary-condition`. Use `x.prop` when the value is provably non-null at the read site.
  - HTTP requests never call `adapter.upgrade()`, so a separate `httpAdapter` with a no-op `upgrade()` is cleaner than sharing a single adapter whose `upgrade()` accesses mutable state.

## 2026-03-01 - US-011
- **What was implemented**: Fixed `export-har` to not lose HTTP request data when the WebSocket frame fetch fails.
  1. `export-har.ts:handler` — Changed both `getNetworkRequests` and `getWebSocketFrames` dispatches to omit the `clear` flag during the initial fetch phase. After all fetches succeed, if `args.clear` is true, separate clear dispatches are issued (with `clear: true`) to flush the extension buffers.      - "both buffers are cleared after all fetches succeed when clear: true with WebSocket frames" — verifies the exact 4-call sequence: fetch HTTP, fetch WS, clear HTTP, clear WS.
     - "HTTP buffer is cleared after fetch succeeds when clear: true without WebSocket frames" — verifies the 2-call sequence when `includeWebSocketFrames` is false.
- **Files changed**: `platform/mcp-server/src/browser-tools/export-har.ts`, `platform/mcp-server/src/browser-tools/export-har.test.ts`
- **Learnings for future iterations:**
  - When two independent side effects (clear HTTP buffer, clear WS buffer) depend on both reads succeeding, separate the read and write phases: read all → verify success → write all. Never interleave reads and writes.
  - The extension's `getNetworkRequests` and `getWebSocketFrames` dispatches accept `{ tabId, clear?: boolean }` — omitting `clear` defaults to no-clear; passing `clear: true` clears the buffer and returns (now-empty) data in the same call.

  - The `export-har.test.ts` mock pattern uses `vi.hoisted` + `vi.mock('../extension-protocol.js')` with `mockDispatchToExtension.mockResolvedValueOnce` for sequential call expectations.
