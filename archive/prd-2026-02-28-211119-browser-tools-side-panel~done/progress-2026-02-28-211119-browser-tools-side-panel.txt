## Codebase Patterns
- platform/shared/src/index.ts is the single file for all shared types — no separate files per domain
- ConfigStateResult is the wire type for config.getState response; add new fields here to extend the protocol
- ConfigStateBrowserTool/ConfigSetBrowserToolEnabledParams follow the same naming convention as ConfigStatePlugin/ConfigSetToolEnabledParams
- handleConfigGetState is in platform/mcp-server/src/extension-handlers.ts; uses state.cachedBrowserTools (CachedBrowserTool[]) and isBrowserToolEnabled from state.js
- To test handlers that call sendToExtension, mock state.extensionWs = { send: msg => messages.push(msg), close: () => {} }
- CachedBrowserTool has a tool: BrowserToolDefinition field; in tests use `null as never` to avoid importing BrowserToolDefinition
- McpCallbacks interface in extension-handlers.ts is used across 3 test files (extension-handlers.test.ts, extension-protocol.test.ts, mcp-setup-dispatch.test.ts) — adding a new field requires updating noopCallbacks in all of them
- saveBrowserToolPolicy follows the same read-modify-write pattern as saveToolConfig in config.ts — both use configWriteMutex serialization
- config.setBrowserToolEnabled handler sends a plugins.changed notification after toggling, so the side panel re-fetches state
- BrowserToolsCard follows PluginCard pattern: AccordionPrimitive for custom header, optimistic toggle with rollback, error alert auto-dismiss
- Browser tool active tool key format is `browser:${toolName}` (e.g., `browser:browser_list_tabs`) — MCP server sends plugin='browser'
- BrowserToolState is a type alias for ConfigStateBrowserTool in bridge.ts; no displayName/icon fields on the wire type, so they are derived
- toDisplayName strips 'browser_' prefix and title-cases the remaining words
- ToolRow's icon prop accepts lucide icon names as strings; 'globe' works for all browser tools
- knip tracks `src/**/*.stories.tsx` as entry points — a new component needs a stories file to avoid being flagged as unused
- When adding new required props to a component, update ALL stories files that use it (both the component's own stories and SidePanel.stories.tsx / SearchResults.stories.tsx)
- ConfigStateResult.browserTools is non-optional (`ConfigStateBrowserTool[]`), so `?? []` triggers @typescript-eslint/no-unnecessary-condition
- Active tool keys for browser tools use `browser:` prefix; loadPlugins must preserve these in the activeTools cleanup filter
- BrowserToolsCard renders inside its own `<Accordion type="multiple">` wrapper, separate from the PluginList Accordion
- SearchResults matches browser tools via toDisplayName, tool.name, and tool.description — also matches 'browser tools' header text
- E2E tests that checked for 'No Plugins Installed' must check for 'Browser Tools' instead (browser tools are always shown when connected)
- E2E tests using `.locator('button[aria-expanded]').first()` or `.locator('.border-border').first()` break when BrowserToolsCard is added — use `.filter({ hasText: 'E2E Test' })` to scope to the plugin card
- `button[data-radix-collection-item]` locator in strict mode fails when multiple accordion buttons exist — use a specific filter or remove redundant assertions
