## Codebase Patterns
- confirmation-badge.ts uses module-level Set (clearedConfirmationIds) and Map (confirmationTimeouts) for idempotency and background timeout tracking
- clearConfirmationBadge is idempotent per id — returns early if id already in clearedConfirmationIds
- The side panel ALWAYS pairs clearConfirmationBadge(id) + clearConfirmationBackgroundTimeout(id) in that order
- clearConfirmationBackgroundTimeout cancels the background timeout; if this fires before the bg timeout, the background timeout never fires
- handleBrowserExecuteScript uses Promise.race between an IIFE and a timeout; cancellation flag ({ value: false }) in closure lets the IIFE exit early when outer timeout wins
- In extension-commands.test.ts, accessing mock.calls elements requires double-casting: `(calls[i] as unknown[])[0] as Record<string, unknown>` to avoid no-unsafe-assignment
- expect.stringContaining()/expect.any() in object literals require `as string`/`as Type` cast to avoid no-unsafe-assignment in this codebase's strict ESLint config
- Tests with fake timers need vi.useFakeTimers() in beforeEach and vi.useRealTimers() in afterEach; use vi.advanceTimersByTimeAsync(ms) for timer-driven async code
