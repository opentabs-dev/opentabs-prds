## Codebase Patterns
- The MCP server binds to 127.0.0.1 (IPv4 only), not localhost. E2E test fetch calls should always use 127.0.0.1 to avoid IPv6 resolution issues on macOS where localhost may resolve to ::1 first.
- The dev proxy (dev-proxy.ts) also binds to 127.0.0.1 and proxies to the worker on 127.0.0.1. All internal proxy connections use 127.0.0.1.
- DEFAULT_HOST in platform/shared/src/constants.ts is set to '127.0.0.1' explicitly to avoid IPv6 issues.
- The isLocalhostHost() function in http-routes.ts allows localhost, 127.0.0.1, ::1, and ::ffff:127.0.0.1 as valid Host headers.

## 2026-04-01 - US-001
- What was implemented: Changed all fetch calls in e2e/tool-http-api.e2e.ts from `http://localhost:${port}` to `http://127.0.0.1:${port}` to match the MCP server's bind address. This eliminates IPv6 resolution issues on macOS where `localhost` resolves to `::1` first, causing ECONNREFUSED before the Happy Eyeballs fallback to IPv4.
- Files changed:
  - e2e/tool-http-api.e2e.ts (3 fetch URL changes: fetchWithAuth helper and 2 direct fetch calls in 401 auth tests)
- **Learnings for future iterations:**
  - On macOS, `localhost` resolves to both `::1` and `127.0.0.1`. When the server binds to `127.0.0.1` only, connecting via `localhost` causes an IPv6 ECONNREFUSED before falling back to IPv4 via Happy Eyeballs (RFC 8305). This can cause intermittent failures or slower connections on macOS CI.
  - Always use `127.0.0.1` directly in E2E test HTTP calls, matching the server's bind address. This pattern is already used in the dev proxy (proxyHttp, reinitializeSessions) and other E2E fixtures (fetchHealth, waitForHealth).
  - The macOS CI job (ci.yml) runs on `macos-latest` without `xvfb-run` (macOS has a native display server). It does NOT set CI=1 for Playwright workers — wait, it does set `CI: '1'` in the env.
