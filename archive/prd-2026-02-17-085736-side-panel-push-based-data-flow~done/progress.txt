## Codebase Patterns
- `forwardToSidePanel` from `messaging.ts` wraps messages in `sp:serverMessage` type for the side panel listener in App.tsx
- `sendToServer` sends JSON-RPC over WebSocket to MCP server; `forwardToSidePanel` sends to the side panel via chrome.runtime.sendMessage
- Tab state changes flow through three call sites: `checkTabStateChanges` (tab events), `sendTabSyncAll` (connect/reconnect), `handlePluginUpdate` (hot reload)
- The side panel message listener in App.tsx matches on `data.method` to route notifications (tab.stateChanged, plugins.changed, tool.invocationStart/End)
- `SIDE_PANEL_METHODS` in message-router.ts controls which server messages are forwarded to the side panel by the generic `handleServerMessage` router
- The `lastKnownState` Map in tab-state.ts is the authoritative cache for per-plugin tab states in the extension
- `getPluginTabState(pluginName)` in tab-state.ts reads from `lastKnownState`, returning 'closed' as default — use this to build ConfigStatePlugin objects with tab state
- The validated plugin payloads in handleSyncFull (`metas` array) have the same shape as ConfigStatePlugin except for `tabState` and `displayName` defaulting — map with `displayName: m.displayName ?? m.name` and `tabState: getPluginTabState(m.name)`
- New push notification methods: `plugins.state` (full list), `plugin.updated` (single update) — both must be in SIDE_PANEL_METHODS
