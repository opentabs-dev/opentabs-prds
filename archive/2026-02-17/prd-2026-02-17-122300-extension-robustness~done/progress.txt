## Codebase Patterns
- The keepalive alarm handler in background.ts fires periodically (KEEPALIVE_INTERVAL_MINUTES) and is the right place to add periodic health checks
- ensureOffscreenDocument() is idempotent — it attempts chrome.offscreen.createDocument and catches the error if one already exists
- The browser extension uses a creatingOffscreen promise guard to prevent concurrent creation attempts
- Extension lifecycle events (onInstalled, onStartup, top-level) all call ensureOffscreenDocument() + setupKeepaliveAlarm() + reinjectStoredPlugins()
- The offscreen ws:setUrl handler has three branches: active ws, pending reconnect timer, and neither (backoff exhausted). All three must be covered for URL changes to always trigger reconnection.
- The codebase pattern for chrome.runtime.sendMessage is promise-style `.catch()` (see messaging.ts). The bridge.ts sendRequest was the last callback-style holdout. getConnectionState still uses callback-style because it needs the response value.
- ESLint enforces `@typescript-eslint/use-unknown-in-catch-callback-variable` — catch callbacks must type the error as `unknown`, not `Error`.
- Side panel E2E tests use manual setup (not fixtures) for side-panel-data-flow tests: MCP server + test server + extension context + adapter symlink, with try/finally cleanup.
- When the plugin is in 'unavailable' state, `TabStateHint` renders "Log in to {displayName}" which can cause `getByText(displayName)` to match multiple elements. Use `locator('button[aria-expanded]').first()` to target the plugin card button directly.
- To trigger a tab state recheck after toggling auth, reload the app tab — the `status=complete` event fires `checkTabStateChanges` in the background script.
- Tab state CSS classes: ready → `bg-emerald-400`, unavailable → `bg-amber-400`, closed → `bg-red-400`.
- Bun fires the WebSocket close handler synchronously during `ws.close()`. In `createHandleWsOpen`, `state.extensionWs` must be assigned to the new WS BEFORE calling `.close()` on the old one, otherwise the close handler rejects pending dispatches incorrectly.
- To test the server-side DISPATCH_TIMEOUT_MS (30s), replace the extension's WS with a fake client mid-dispatch so the response has no path back. Use the test server's slow mode (60s) to ensure the adapter is still blocked when the WS is replaced. Use a raw HTTP fetch with a 45s timeout (not the standard mcpClient which has 30s).
- Opening a raw WebSocket to the MCP server replaces the extension's connection. The extension reconnects quickly (~1s), so any raw WS tests must complete their sends rapidly without inter-message waits.
