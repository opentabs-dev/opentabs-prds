## Codebase Patterns
- Plugin auth extraction runs in the content script (injected IIFE) context, so `document`, `localStorage`, and `window` are all available as same-origin page APIs
- The `getAuth()` function chains multiple extraction methods via `??` (nullish coalescing) ‚Äî add new methods to the end of the chain
- Plugin builds (`cd plugins/slack && bun run build`) run `tsc` then `opentabs build` which bundles `dist/adapter.iife.js` ‚Äî the MCP server file watcher picks up changes automatically
- Plugins are standalone projects: their code is NOT covered by root `bun run lint`/`bun run type-check`; use plugin-local commands to verify
- All extraction methods must be wrapped in try/catch returning null on failure ‚Äî the adapter runs in arbitrary page contexts
- Side panel `setPlugins` updater runs synchronously inside React's batch ‚Äî if `prev` is `[]`, mapping over it is a no-op and notifications are silently lost; buffer early notifications in a ref and replay after fetchConfigState resolves
