## Codebase Patterns
- `forwardToSidePanel` from `messaging.ts` wraps messages in `sp:serverMessage` type for the side panel
- Side panel filters incoming messages by `SIDE_PANEL_METHODS` set in `message-router.ts` — only `tab.stateChanged`, `tool.invocationStart`, `tool.invocationEnd`, `plugins.changed` are forwarded
- Tab state is cached in `lastKnownState` map in `tab-state.ts` — redundant notifications are suppressed
- `sendToServer` and `forwardToSidePanel` are both fire-and-forget (catch errors silently)
- Three places emit tab.stateChanged: `checkTabStateChanges`, `sendTabSyncAll` (initial sync), `handlePluginUpdate`
- Side panel data flow uses three patterns: (1) server-originated state uses invalidate + re-fetch via `loadPlugins`/`fetchConfigState`, (2) extension-originated tab state is pushed directly via `forwardToSidePanel`, (3) tool toggles use optimistic local `setPlugins` updates
- ESLint import-x/order: type imports from local modules must come before type imports from external packages
- Playwright: use `getByText('X', { exact: true })` for text matching — `locator('text=X')` does substring matching and causes strict mode violations (e.g., "Connected" matches "Disconnected")
- E2E side panel tests use `getExtensionId`, `openSidePanel`, `setupAdapterSymlink` helpers (currently duplicated in each test file)
- `startMcpServer(configDir, hot, explicitPort)` supports explicit port for server restart tests
- `forwardToSidePanel` (chrome.runtime.sendMessage) is unreliable in Playwright for side panels opened as regular pages. Use /health endpoint + side panel reload as workaround.
- /health endpoint: `pluginDetails` is the array, `plugins` is just a count number
- background.ts: `injectPluginsIntoTab` must complete BEFORE `checkTabStateChanges` on `status:complete` events (injection → state check sequence)
- App.tsx `handleNotification` handles `tab.stateChanged`, `tool.invocationStart`, `tool.invocationEnd` — shared between `sp:serverMessage` and `ws:message` paths
- `testServer.setSlow(ms)` adds configurable delay to all tool responses — useful for animation/timing E2E tests
- `createMcpClient(port)` from fixtures.ts creates a direct MCP client for calling tools in E2E tests
