## Codebase Patterns
- E2E tests use dev-only `POST /__test/set-outdated` endpoint to inject fake outdated plugin data into MCP server state for testing update UI
- E2E tests use dev-only `POST /__test/simulate-update` endpoint to simulate a successful plugin update by mutating the version in the registry
- The `buildConfigStatePayload` in `extension-handlers.ts` maps `state.outdatedPlugins` entries to plugin `update` fields using `npmPackageName` as the join key
- Local plugins DO have `npmPackageName` (from `pkg.name` in package.json), so they can be matched against `outdatedPlugins` entries
- E2E test cleanup uses `finally` blocks; `cleanupDir` from `launchExtensionContext` must be checked for truthiness before `rmSync`
- Clicking Update on a local plugin fails because `npm update -g` can't find the package globally — this naturally tests the error path
- The error alert on a plugin card uses `[role="alert"]` and appears when `actionError` prop is set via `pluginErrors` map in App.tsx
- The PluginRegistry is "immutable" by convention but the JavaScript objects inside are mutable — test endpoints can mutate `plugin.version` directly
