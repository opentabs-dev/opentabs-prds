## 2026-03-30 03:50 UTC - US-003
- Replaced `preToggleRef` (shared `WireToolDef[]` ref) with `pendingToolRollbacks` (`Map<number, { toolName, prev }>`)
- Replaced `prePluginPermRef` (shared `ToolPermission` ref) with `pendingPluginRollbacks` (`Map<number, ToolPermission>`)
- Each optimistic update now stores its own rollback state keyed by the toggle counter version
- On failure, each operation rolls back only its specific tool/permission change, independent of other in-flight operations
- `.finally()` cleans up the Map entry after each operation completes to prevent unbounded growth
- Removed the `toggleCounter.current === myVersion` staleness guard — per-operation rollback makes it unnecessary
- Files changed: platform/browser-extension/src/side-panel/components/PluginCard.tsx
- **Learnings for future iterations:**
  - The core bug: shared refs (`preToggleRef`, `prePluginPermRef`) are overwritten by each new operation. When operation A fails after operation B has already overwritten the ref, A's rollback either doesn't happen (version check) or uses B's snapshot (which includes A's optimistic update).
  - The fix pattern: store per-operation rollback data in a Map<operationId, rollbackData>. Each operation captures only what it needs to undo (the specific tool name + its previous permission), not the entire state array. This makes rollbacks composable and independent.
  - The `toggleCounter` is still useful as a unique key generator for Map entries, but it should NOT be used as a staleness guard (`if (counter === myVersion)`) for rollback — stale operations still need their rollback applied.
