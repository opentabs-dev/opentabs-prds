## Codebase Patterns
- Platform CLI source lives in `platform/cli/src/commands/` with one file per command (start.ts, stop.ts, config.ts, etc.)
- `printMcpClientConfigs` is shared between `start --show-config`, `config show --show-secret`, and first-time setup — `primaryOnly=false` shows all 4 configs, `primaryOnly=true` shows only Claude Code + a "For other MCP clients" hint; after US-003, first-run uses `primaryOnly=false`
- The CLI uses `picocolors` (imported as `pc`) for terminal coloring
- `@opentabs-dev/shared` exports cross-platform helpers like `platformExec`, `isWindows`, `toErrorMessage`
- The `handleStart` function in start.ts handles both foreground and `--background` modes
- Background mode spawns a detached child process writing to a log file; foreground mode uses `teeStream` to pipe output to both terminal and log file

- Files changed: `platform/cli/src/commands/start.ts`
- Changes:
  - Extracted `printStartupHeader()` and `printSetupHints()` closures from inline code so the header can be printed conditionally
  - Moved header + hints printing to AFTER spawn + health check in the background branch (prevents showing success-like output before confirming server is alive)
  - Added 2-second post-spawn health check: listens for `exit` event on the child process; if it exits within 2s, prints warning with log path
  - Wrapped PID file `writeFile` in try-catch with user-friendly "Failed to write PID file: <reason>" message
  - The foreground path still prints header/hints in the same position as before (just refactored into the closures)
- **Learnings for future iterations:**
  - `child.pid` is undefined when spawn itself fails (e.g., binary not found) — checking it immediately after spawn is the right guard
  - `child.unref()` makes the child not keep the parent's event loop alive, but you can still listen on `child` events before the parent exits
  - The `resolve` shadow in Promises using `node:path`'s `resolve` is a common pitfall — inner Promise `resolve` takes precedence
