## Codebase Patterns
- E2E tests create temp dirs with `mkdtemp` using prefixes like `opentabs-e2e-config-`, `opentabs-e2e-plugin-copy-`, and `opentabs-e2e-` (for Chromium user-data and extension dirs)
- MCP servers are spawned with `node /path/to/dist/dev-proxy.js` (hot) or `node /path/to/dist/index.js --dev` (non-hot); the configDir is passed via `OPENTABS_CONFIG_DIR` env var, NOT as a command line argument
- Chromium instances have `--user-data-dir` and `--load-extension` paths containing `opentabs-e2e-`, making bare `pgrep -f "opentabs-e2e-"` match browsers
- On macOS, `pgrep -f` may inspect env vars (via `sysctl KERN_PROCARGS2`), making it match MCP server processes via their env var; on Linux it only searches the argv[], not env vars
- `e2e/orphan-guard.ts` handles MCP server cleanup via parent-process polling â€” the global teardown is a last-resort safety net for any remaining processes
- Phase 1 (build + type-check + lint + knip + test) is sufficient for stories with `e2eCheckpoint: false`
- In E2E tests, always declare `let server: McpServer | undefined` before the try block and assign inside it; use `server?.kill()` in finally to prevent cleanup skips when startMcpServer rejects
- ESLint rule `import-x/consistent-type-specifier-style` requires top-level `import type { Foo }` rather than inline `type Foo` within a value import; and `import-x/order` requires the type import to follow all node: imports in the same group

  - The global teardown is mainly useful for catching Chromium orphans and (on macOS) MCP server orphans via env var matching
