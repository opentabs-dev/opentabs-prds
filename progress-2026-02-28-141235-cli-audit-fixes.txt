## 2026-02-28 - US-002
- Root cause: `createReadStream` with `encoding: 'utf-8'` uses an internal `StringDecoder` scoped to one stream instance. When a multi-byte UTF-8 char straddles two read cycles (the file's byte count lands mid-char), the first stream's decoder flushes incomplete bytes as U+FFFD on `end`, and the second stream starts fresh â€” producing a second replacement char for the continuation bytes.
- Fix: removed `encoding: 'utf-8'` from `createReadStream` and introduced a persistent `let decoder = new StringDecoder('utf-8')` in `followFile`. Each `data` event calls `decoder.write(chunk)` (Buffer path); incomplete sequences are buffered inside the decoder and emitted as complete characters when the remaining bytes arrive in the next read cycle.
- Decoder is reset (new instance) on file truncation and on stream error to avoid stale state after position resets.
- TypeScript's `ReadStream` types require `(chunk: string | Buffer) => void`; used `Buffer.isBuffer(chunk)` runtime guard to route Buffers through `decoder.write` and strings (should not occur without encoding, but satisfies the type) through directly.
- Added test: "does not produce replacement characters when a multi-byte UTF-8 character spans read cycles" â€” emits the first 2 bytes of đźŤŽ (0xF0 0x9F) in cycle 1 and the last 2 bytes (0x8D 0x8E) in cycle 2; asserts stdout output equals `'đźŤŽ'` with no `\uFFFD`.
- Files changed: `platform/cli/src/commands/logs.ts`, `platform/cli/src/commands/logs.test.ts`
- **Learnings for future iterations:**
  - The across-cycle split (file size lands mid-char) is a different bug from the within-stream split (chunk boundaries inside one stream). Node's stream encoding handles within-stream splits; a persistent `StringDecoder` is needed for across-cycle splits.
  - Never call `decoder.end()` between read cycles â€” it flushes incomplete bytes as replacement chars. Only instantiate a new decoder on truncation or error.
  - The `process.stdout.write` spy correctly captures empty-string calls (when decoder buffers a partial sequence); joining all call args gives the total output.

## 2026-02-28 - US-003
- Root cause: `logStream.end()` was called but not awaited before `process.exit(exitCode)`. Node.js streams buffer writes internally; calling `process.exit()` immediately terminates the process before the OS flushes those buffers, truncating the final log bytes.
- Fix: replaced `logStream.end()` with `await new Promise<void>(resolve => logStream.end(() => resolve()))`. The callback form of `.end()` fires only after the stream is fully flushed to the underlying file descriptor.
- Files changed: `platform/cli/src/commands/start.ts` (line 349, one-line change)
- No test file exists for `start.ts`; the fix is a correctness guarantee for process shutdown ordering that can't easily be unit-tested without spawning real processes.
- **Learnings for future iterations:**
  - The callback form of `WriteStream.end(callback)` is the idiomatic way to await flush; wrapping in `new Promise<void>` makes it awaitable in async functions.
  - `process.exit()` bypasses Node.js's normal stream drain â€” always flush streams explicitly before calling it.
