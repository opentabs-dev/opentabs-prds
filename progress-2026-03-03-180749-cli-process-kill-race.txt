## Codebase Patterns
- CLI commands `stop.ts` and `update.ts` both duplicate `isProcessAlive` and `waitForExit` helpers (intentionally not shared yet)
- `isProcessAlive` uses `process.kill(pid, 0)` with try/catch — SIGTERM calls should follow the same defensive pattern
- Node.js `process.kill()` throws with `code: 'ESRCH'` when the target PID no longer exists

## 2026-03-04 - US-001
- Wrapped `process.kill(pid, 'SIGTERM')` in try/catch in both `stop.ts` and `update.ts`
- In `stop.ts`: on ESRCH, cleans up PID file and prints "Server stopped." (early return)
- In `update.ts`: on ESRCH, skips `waitForExit` (checks `isProcessAlive` first) and proceeds to restart flow
- Both files re-throw non-ESRCH errors (e.g., EPERM) so they surface to the user
- Files changed: `platform/cli/src/commands/stop.ts`, `platform/cli/src/commands/update.ts`
- **Learnings for future iterations:**
  - The race window in `stop.ts` is wider than `update.ts` because `stop.ts` does a 2s health check between the alive check and the kill
  - The `err.code === 'ESRCH'` check requires `'code' in err` guard for TypeScript narrowing since `Error` doesn't have a `code` property by default
