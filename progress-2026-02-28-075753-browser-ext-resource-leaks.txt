## Codebase Patterns
- network-capture.ts: CDP event dispatch uses a flat if/else if chain in `chrome.debugger.onEvent.addListener`; new event types go there
- network-capture.test.ts: Chrome APIs are stubbed at module-level before dynamic import; `capturedOnEventListener` captures the registered handler for test invocation
- The `wsFramesByRequestId` Map stores requestId → WebSocket URL for active connections; entries added on `webSocketCreated`, deleted on `webSocketClosed` (cleanup)
- `getWsFrames` is exported from network-capture.ts and is importable in tests
- tab-state.ts: `withPluginLock` is NOT exported; test it indirectly via `updateLastKnownState`, `checkTabChanged`, `checkTabRemoved`
- ESLint rule `@typescript-eslint/no-floating-promises` IS configured; use `void` operator for intentional fire-and-forget promises
- side-panel-toggle.ts: MV3 service workers can miss events while suspended; validate window existence with `chrome.windows.get(windowId)` in a try/catch before trusting in-memory Set state
- `void (async () => { ... })()` pattern is the correct way to run async logic inside a synchronous Chrome event listener without triggering floating-promise lint errors
- iife-injection.ts: `injectLogRelay` ISOLATED world func uses a `win[guard]` idempotency check; on re-injection, clear the nonces Set before adding the new nonce (old nonces are invalid after adapter replacement)
- iife-injection.test.ts: to test ISOLATED world funcs, set `(globalThis as Record<string, unknown>).window = {}` before running them (funcs reference `window` which is undefined in Node.js); clean up with `afterEach`
