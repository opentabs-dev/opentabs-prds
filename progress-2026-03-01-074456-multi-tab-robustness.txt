## Codebase Patterns
- tabId extraction pattern: `typeof rawTabId === 'number' && Number.isInteger(rawTabId) && rawTabId > 0 ? rawTabId : undefined` â€” use `typeof` first so TypeScript narrows to `number` before the arithmetic comparison
- There are 4 tabId extraction points across MCP server and browser extension: mcp-tool-dispatch.ts, tool-dispatch.ts, resource-prompt-dispatch.ts (x2), useServerNotifications.ts
- In resource-prompt-dispatch.ts, introduce `const rawTabId = params.tabId;` to avoid accessing params.tabId multiple times in the ternary
- Unit tests for tabId validation: mcp-tool-dispatch.test.ts uses `test.each` near line 1141; tool-dispatch.test.ts uses `test.each` near line 384
- The tool-dispatch.test.ts pattern: set up plugin in storage + mock tabs.query([]) â†’ dispatch with invalid tabId â†’ expect -32001 error (fallback dispatch with no tabs)
- E2E pattern for testing unavailable tab: `setAuth(false)` + `page.reload()` + wait for adapter re-injection + poll `plugin_list_tabs` until `ready:false`
- After `testServer.reset()` auth is restored to true â€” must re-call `setAuth(false)` if still testing unavailable state
- To create per-tab mixed readiness: use `page.route('**/api/auth.check', ...)` to intercept auth.check on one page, then `page.reload()` â€” the fresh adapter's `isReady()` calls the intercepted endpoint; more reliable than monkey-patching + hash navigation
- `waitForTabCount` only checks `tabs.length >= count`, not readiness â€” use `waitFor` with `tabs.every(t => t.ready)` when both count AND readiness matter
- Tab close tracking: after closing all tabs, `plugin_list_tabs` reports `state:'closed'` with empty tabs array
