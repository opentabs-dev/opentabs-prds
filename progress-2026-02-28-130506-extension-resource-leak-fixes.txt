## Codebase Patterns
- confirmation-badge.ts: The `clearedConfirmationIds` set guards against double-decrements. The pattern is: add to the set when clearing, then prune from the set when the entry is no longer needed for idempotency (i.e., when no background timeout can race). The bg timeout callback always prunes after firing; the fix also prunes in `clearConfirmationBadge` when `confirmationTimeouts` doesn't have the id.
- Production call order in background-message-handlers.ts: `clearConfirmationBackgroundTimeout(id)` is always called BEFORE `clearConfirmationBadge(id)` in both `handleSpConfirmationResponse` and `handleSpConfirmationTimeout`. Tests should test this same order.
- Browser extension tests use `vi.useFakeTimers()` and reset state via `clearAllConfirmationBadges()` in beforeEach.
- network-capture.ts: Use `vi.useFakeTimers()` + `vi.advanceTimersByTime(n)` to test that periodic prune intervals fire and clean up entries. Use `vi.setSystemTime(Date.now() + n)` to advance only the clock (not fire timers) when testing inline prune logic triggered by a specific event.
- network-capture.ts: WS_TTL_MS (5 min) is separate from PENDING_REQUEST_TTL_MS (60 s) because WebSocket connections are long-lived; pruning them too aggressively would drop active connections.
- When adding a new map to CaptureState, ensure it is: (1) initialized in startCapture, (2) populated and deleted in the appropriate event handlers, (3) pruned in the periodic interval, (4) pruned in any inline prune site, (5) cleared in stopCapture, and (6) cleared in any clear=true read operation (e.g., getWsFrames).
- network-capture.ts: The `pendingCaptures` Map<number, Promise<void>> serializes concurrent startCapture calls. The pattern is: check pendingCaptures first (return existing promise if present), then check captures (throw if already active), wrap async work in an IIFE, store in pendingCaptures, await with a finally block that deletes from pendingCaptures. The IIFE variable shadowing the outer loop variable `pending` was renamed to `pendingReq` to avoid lint confusion.
- For concurrent async tests: firing both calls synchronously before any await resolves ensures the second call sees the pendingCaptures entry. Use `Promise.allSettled` (not `Promise.all`) to test failure cases — both promises share the same underlying rejection, and `Promise.all` would cause unhandled rejection noise.
