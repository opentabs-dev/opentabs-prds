## Codebase Patterns
- Scripts in `scripts/` are type-checked by `tsconfig.scripts.json` with `noEmit: true` ŌĆö use extensionless imports (not `.ts` or `.js`) since `moduleResolution: "bundler"` is configured
- Knip treats all `scripts/*.ts` files as entry points, so exports from new script files are considered used
- The `ws` package + `@types/ws` are available as root dependencies
- The dev orchestrator (`scripts/dev.ts`) uses top-level await and runs via `tsx`
- lint-staged runs on commit; verify unstaged changes survived after each commit with `git diff --stat && git stash list`
- Extension dist/ layout: `dist/side-panel/` = side panel, `dist/background.js` + `dist/offscreen/` = background/offscreen, other `.js` in dist root = shared code
- The dev orchestrator now has targeted rebuild: `buildSidePanel()` and `buildBackground()` for incremental builds, `buildExtension()` for initial full build
- `devReloadServer.broadcast('side-panel')` for UI-only reloads, `broadcast('extension')` for full extension reloads
- Plain JS build assets in `platform/browser-extension/src/dev/*.js` are read via `readFileSync` at build time and injected via esbuild `banner` ŌĆö not imported, so must be ignored by knip
- esbuild `banner` option injects raw text without `define` replacements ŌĆö use string substitution for build-time constants in banner content
- `build-side-panel.ts` checks `process.env.OPENTABS_DEV === '1'` to conditionally include dev reload clients (set by the dev orchestrator)
- `build-extension.ts` follows the same pattern: `isDev` check ŌåÆ `readFileSync` ŌåÆ string substitution ŌåÆ esbuild `banner` option (background bundle only, not offscreen)
- Background reload client uses `addEventListener` (service worker context) instead of `ws.onopen` etc. (both work, but addEventListener is idiomatic for service workers)
- Background client clears `wsConnected` session storage before `chrome.runtime.reload()` ŌĆö mirrors `handleExtensionReload` in `message-router.ts`
