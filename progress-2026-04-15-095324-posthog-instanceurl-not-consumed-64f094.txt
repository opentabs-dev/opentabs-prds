## Codebase Patterns
- PostHog plugin uses relative API paths (`/api/environments/${teamId}/...`, `/api/projects/${teamId}/...`) that resolve against the current page origin via `fetchJSON`
- The `instanceUrl` configSchema field (type `'url'`) is consumed by the platform to derive URL match patterns for adapter injection into self-hosted tabs — it is NOT used for API URL construction
- The adapter runs in the MAIN world of the matched tab, so relative paths correctly resolve to the tab's origin (whether cloud or self-hosted)
- Adding `getConfig('instanceUrl')` as an API base URL would break multi-instance usage: if a user has both cloud and self-hosted tabs open, the adapter on cloud tabs would incorrectly send API calls to the self-hosted instance

## 2026-04-15 - US-001
- Clarified the `instanceUrl` configSchema description in both `src/index.ts` and `package.json`
- Old: "The URL of your self-hosted PostHog instance (e.g., https://posthog.example.com). Leave empty to use PostHog Cloud."
- New: "The URL of your self-hosted PostHog instance. Used to inject the adapter into your instance — leave empty to use PostHog Cloud."
- Files changed: `plugins/posthog/src/index.ts`, `plugins/posthog/package.json`
- **Learnings for future iterations:**
  - The PRD notes explicitly asked to evaluate whether `getConfig('instanceUrl')` should be used as an API base URL or if the description should be clarified instead. The relative-URL approach is correct because the adapter is injected into the correct tab by the platform, and relative paths resolve against the tab's origin.
  - Using absolute URLs from `getConfig('instanceUrl')` would cause cross-instance contamination when users have multiple PostHog instances open simultaneously.
  - The GitLab plugin uses `getConfig('instanceUrl')` for API base URL construction, but that's because GitLab's API uses absolute URLs (`https://gitlab.com/api/v4/...`). PostHog's adapter uses relative paths, making the approaches fundamentally different.
