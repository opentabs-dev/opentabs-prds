## Codebase Patterns
- MCP server version is read from package.json at startup in `platform/mcp-server/src/version.ts` and exported as `version`. Import alphabetically: comes after `state.js`.
- `ConfigStateResult` in `platform/shared/src/index.ts` is the shared type for `config.getState` responses between MCP server and Chrome extension.
- Extension handler tests use a mock WsHandle pattern (`createMockWs`) that captures messages as JSON strings; parse with `JSON.parse` to inspect result structure.
- ESLint enforces `import-x/order` ŌĆö imports must be in alphabetical order by path within each import group (case-insensitive: `./BrowserToolsMenu.js` < `./retro/Accordion.js` because `b` < `r`).
- BrowserToolsMenu is the counterpart to PluginMenu ŌĆö same wrapper structure (div + onClick/onKeyDown stopPropagation + role=presentation), same button style (h-6 w-6, hover:bg-muted/50), same `className="flex shrink-0 items-center px-1"` on the card.
- Some component files use `.js` import extensions (BrowserToolsCard.tsx) while others don't (PluginMenu.tsx) ŌĆö both pass lint. TypeScript resolves `.tsx` when you write `.js` in imports.
- When adding props to JSX components in App.tsx that would exceed the print width, prettier requires multi-line format ŌĆö all props on separate lines with closing `/>` on its own line, indented to match the opening tag's content indent.
