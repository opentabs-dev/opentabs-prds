## Codebase Patterns
- E2E tests that need custom setup (non-fixture) follow the pattern in auth-bootstrap.e2e.ts: createTestConfigDir() → startMcpServer() → launchExtensionContext() with try/finally cleanup
- launchExtensionContext(port, secret?) creates a copy of the extension patched for the given port; omitting secret means no auth.json is written
- openSidePanel(context) from helpers.ts opens the side panel as a chrome-extension:// page
- Side panel empty states: "Authentication Failed" (wrong secret), "Cannot Reach MCP Server" (server down), "No Plugins Installed" (no plugins)
- Two MCP servers sharing the same configDir share the same auth secret (auth.json is in configDir/extension/auth.json)
- NumberStepper input can be targeted with getByLabel('Server port'); use fill() + press('Enter') to commit the value
- The test server's /api/fail `error` field is used by test-api.ts as the ToolError message — it must contain the human-readable message, not the code
